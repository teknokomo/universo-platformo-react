# Source Quality Rubric

Use this rubric when deciding which sources can support a research-before-PLAN artifact.

## Primary Sources

Prefer these whenever available:

- Official documentation
- Official repositories and source code
- Release notes and changelogs
- Standards and specifications
- Academic papers
- Official pricing, security, legal, or compliance pages

Primary sources can support implementation decisions directly when they are current and relevant.

## Secondary Sources

Use these to add context, examples, or implementation experience:

- Vendor engineering blogs
- Maintainer-authored posts
- Credible engineering writeups
- Conference talks with published materials

Secondary sources should not override primary sources when they conflict.

## Tertiary Sources

Use these only as weak signals or leads:

- Forums
- Reddit
- Social media
- Personal blogs
- Unverified tutorials

Tertiary sources require corroboration before they influence a plan.

## Freshness Rules

- Same-day research is preferred for AI model/API behavior, package versions, pricing, security guidance, legal/compliance claims, and current product capabilities.
- Stable architecture concepts may reuse research up to 30 days old if no freshness concerns are recorded.
- User-provided links must be opened even when a related prior artifact exists.

## Minimum Evidence Policy

- Open every user-provided URL.
- For non-trivial architecture or workflow decisions, include at least two additional authoritative sources.
- Record conflicts and uncertainty explicitly.
- Do not turn weak evidence into confident claims.
