## Research Question

What storage model should Universo Platformo use for PlayCanvas projects authored through the already attached PlayCanvas Editor package, so scenes, assets, script references, generated artifacts, snapshots, metahub copy, and runtime publication can be planned without mixing authoring-only Editor state with published application runtime state?

## Key Findings

-   The brief is correctly scoped as a storage-model decision before the Editor bridge/adapter. The repository already has an artifact-only Editor host and package settings; it still has no canonical project store for Editor scenes, assets, script assets, or generated artifacts.
-   The earlier modules-external-files research is now historical context. The repository already implements `ModuleStorageMode`, `ModuleSourceStorage`, file-backed module source paths, snapshot transport, restore checks, copy/delete cleanup, and canonical hashing for modules. The next PlayCanvas storage work should build on that completed module-source layer instead of reopening it.
-   File-backed Modules are intentionally TypeScript/TSX source storage under `modules/...`. `ModuleSourceFileService` enforces `.ts/.tsx`, a `modules/` prefix, a 512 KiB limit, metahub+branch scoped roots, checksum validation, traversal guards, symlink rejection, and tree copy/delete helpers. PlayCanvas project assets need a sibling file-service contract, not reuse of `modules/...` or module-only extension rules.
-   `rel_metahub_packages.config` is already a typed package attachment/display envelope. It should remain an anchor that enables the PlayCanvas Editor authoring surface and optionally points to a project-store record; it should not hold scene JSON, asset indexes, binary metadata, or generated script artifacts.
-   Package attachment persistence already preserves `config` during metahub copy and snapshot restore. That lifecycle support is useful for display settings and a future default-project pointer, but it is also a risk: storing project payload there would silently couple large mutable project data to a config path that is not designed for asset copy/remap, file rollback, or publication hashing.
-   Runtime package publication already filters authoring-only packages: `listPublishedPackages` excludes attachments with empty `source.runtimeTargets`, while metahub snapshots can preserve attachment config. This supports the brief's boundary that authoring-only Editor state must stay out of `_app_packages` unless deliberately projected.
-   Current metahub snapshots use top-level `version: 1` plus `versionEnvelope.snapshotFormatVersion: 3`, and have no PlayCanvas project field. Adding PlayCanvas project storage has two viable paths: add a first-class optional snapshot section such as `playcanvasProjects`, or project only a normalized runtime manifest into existing layouts/modules/settings. PLAN must decide this before migrations and define the snapshot format-version policy.
-   If branch-scoped DB tables are chosen, they must be real system-table version work. New `_mhb_playcanvas_*` tables or columns must update the system table definitions and structure version path for existing branch schemas; baseline-only edits would leave old metahub branches without the storage tables.
-   Canonical publication hashing is a first-order requirement. `publicationSnapshotHash.ts` normalizes known sections; unknown PlayCanvas project fields will not affect release hashes unless the normalizer is updated or the project data is reduced into existing hashed snapshot fields before publication.
-   Application sync uses publication snapshot hash and targeted diff checks to skip work when nothing changed. A PlayCanvas runtime projection therefore needs either a dedicated persistence/diff path or an intentional reduction into existing hashed and persisted fields such as layouts/modules.
-   Snapshot restore for file-backed Modules already validates source content against checksums and writes files after DB inserts with backup/cleanup considerations. PlayCanvas project restore needs equivalent import semantics for project metadata and files: validate hashes before accepting references, restore payload files, clean stale files safely, and fail closed on checksum/path mismatches.
-   Metahub copy is a filesystem concern, not only a database clone concern. The module source file service already has a `copyTree` pattern. PlayCanvas project files will need their own metahub+branch project root and copy/remap/rollback behavior so copied metahubs do not point back to the original asset tree.
-   The existing package artifact host is not a project asset host. It has package-artifact routing, availability, CSP/cache, and iframe concerns. Mutable PlayCanvas project assets and generated script artifacts need a separate route namespace, authorization policy, cache policy, and lifecycle cleanup.
-   The current `playcanvasCanvas` runtime surface is not an Editor project runtime. Its schema stores a small procedural scene object list plus module-backed mount configuration. A future publication step should produce a normalized runtime manifest or widget projection; direct Editor scene JSON consumption should be treated as an exception requiring versioning and isolation proof.
-   The current MMOOMM fixture confirms runtime PlayCanvas is compiled-module driven. It uses `@universo-react/playcanvas-engine` and Colyseus packages through module bundles and widget config. Editor project storage must not make published runtime depend on live authoring files.
-   PlayCanvas Engine docs via Context7 show ESM script assets are loaded as modules, Script classes are registered by `static scriptName` or lower-camel fallback, and script attribute schemas are read from script asset `data.scripts`. Therefore the storage model must persist script asset metadata, not just module source references.
-   Official PlayCanvas docs state ESM scripts use `.mjs`, exported classes, and `static scriptName`, while script attributes can be parsed from ESM JSDoc and surfaced in the Editor. Storage therefore needs to preserve both generated `.mjs` artifacts and parsed attribute schemas, not only the original TypeScript module file.
-   PlayCanvas script attributes are both Editor-facing schema and runtime values. Storage needs separate fields for parsed attribute schema/order/validation results and for per-scene/per-entity script component values.
-   Vendored Editor code reinforces the same structure: script creation rejects same-folder filename collisions, script parsing sends project/branch/asset ids through the realtime pipeline, ESM parsing caches by virtual path and `file.hash`, classic parsing fetches asset file URLs, script components can be URL-based, script panels are keyed by script name, asset inspectors read `data.scripts`, and asset stores build file URLs from asset ids and filenames. A storage model should keep stable asset ids, branch/project scope, virtual paths, file hashes, filenames, script names, parse status, and scene bindings.
-   External official docs and local upstream source do not provide a complete self-hosted Editor backend contract. The storage model should avoid pretending to implement PlayCanvas Cloud semantics; it should define the Universo data envelope that a later adapter can map to Editor API/backend expectations.
-   The most conservative storage shape is hybrid: first-class metahub-scoped project records plus file-backed asset roots. Metadata that needs querying, ownership, copy, snapshot, and package attachment links belongs in DB tables or a typed metadata table; large payloads and binaries belong under a PlayCanvas-specific file-service namespace.
-   A new built-in Platformo entity type is not justified for this slice. The project should prefer existing Object+Components+Modules links for domain metadata and separate PlayCanvas project storage for Editor-specific scene/asset payloads. Only links to Platformo entities should be represented in the metahub entity model.
-   Project storage should survive package version changes, detach/reattach, and compatibility migrations. If project records are anchored by package attachment id alone, detach/recreate flows can orphan content. A stable metahub-scoped project id/codename plus package slug/version compatibility metadata is safer.
-   S3/admin storage provider configuration is related but should remain separate. The project storage model should define provider-neutral metadata (`provider`, `root`, `path`, `hash`, `size`, `mime`, `storageClass`), while actual ENV/admin/S3 setup belongs to a later external storage provider brief.

## Conflicts And Uncertainty

-   The original TZ discusses Editor API, Colyseus integration, AI/MCP authoring, and possibly backend emulation. Those are downstream bridge/adapter concerns. Pulling them into this storage model would make the scope too broad.
-   The current repository has strong module file storage but no generic file-provider abstraction. PLAN must decide whether PlayCanvas asset storage starts as a package-local service modeled after `ModuleSourceFileService` or waits for a broader storage-provider abstraction.
-   It is undecided whether PlayCanvas project metadata should use metahub branch system tables, platform-level tables, JSON envelopes, or a hybrid. Branch system tables align with metahub schema copy/versioning but require system-table migration discipline; platform-level tables are easier to query globally but can drift from branch lifecycle.
-   It is undecided whether generated Editor-consumable `.js`/`.mjs` artifacts should be committed into snapshots, regenerated on import/publish, or both. The safer model is probably both: store source references and checksums plus generated artifacts with validation/rebuild semantics.
-   Direct runtime consumption of Editor scene JSON is uncertain. It may be useful for short-term proof, but it risks coupling `apps-template-mui` to upstream Editor internals. A normalized runtime manifest is safer unless PLAN proves direct JSON is stable and versioned.
-   PlayCanvas Editor API docs are useful but not sufficient to define persistence. The open-source Editor frontend still assumes a larger backend/project/asset ecosystem. Treat Editor API/storage adapter details as a later research topic.
-   Asset type scope is still open. Scripts and scene JSON are likely first; textures/models/audio introduce larger files, MIME validation, quota, thumbnail/processing, and S3/provider concerns.

## Project Implications

-   PLAN should define a `PlayCanvasProject` storage envelope with at least: stable project id/codename, metahub id, branch/schema name, package slug/version compatibility, schema version, settings, scene index, asset index, script asset index, generated artifact index, publication projection metadata, and lifecycle flags.
-   If DB tables are chosen, they must follow the existing backend rules: package-local DDL boundary, schema-qualified parameterized SQL via `DbExecutor`, request-scoped executors for domain routes, and migration registration through the platform migration/system-table path as appropriate.
-   If branch-scoped system tables are chosen, PLAN must include `SYSTEM_TABLE_VERSIONS`, structure semver bump, `SystemTableMigrator`/diff tests, and restore/import behavior for older snapshots that lack PlayCanvas project sections.
-   Do not store project payload in `rel_metahub_packages.config`. That config should at most include a stable project reference or default project selection for the authoring surface.
-   Create a PlayCanvas-specific asset file-service namespace, probably under the same deployment root as module source files but with a different subtree such as `playcanvas-projects/...`. It should copy the security posture of `ModuleSourceFileService` while allowing non-TS asset extensions and larger type-specific limits.
-   Reuse file-backed Modules as source-of-truth for TypeScript/TSX code. PlayCanvas script assets should reference modules through remappable internal ids plus stable logical identifiers (`module codename`, `sourceStorage.path`) and then track generated JS/ESM artifacts separately.
-   Add explicit generated-artifact metadata: artifact id, source module id/codename/path, source checksum, output path, output checksum, script name, module/class export name, ESM/classic type, parse status, and last generated/parsed times.
-   Store script attribute schemas separately from per-scene script instance values. Attribute schemas belong to script asset metadata; values belong to scene/entity script component bindings.
-   PLAN must define the adapter-facing API contract explicitly before bridge implementation. At minimum: read/query operations for `loadProject`, `listScenes`, `listAssets`, `readScene`, `readAssetMetadata`, and `resolveScriptAsset`; mutating operations for `writeScene`, `writeAssetMetadata`, generated-artifact updates, and asset/file writes; lifecycle operations for `publishProjectState`, `exportProjectState`, snapshot import/restore, copy/remap, and delete/cleanup. Mutating and lifecycle operations must participate in checksum validation, optimistic conflict handling, snapshot transport, copy/remap, and publication hash decisions.
-   Define id remapping before implementation. Snapshot import/copy must remap PlayCanvas project ids, scene ids, asset ids, generated artifact ids, module ids, and any Platformo entity links without leaking references to the source metahub.
-   Publication should produce a resolved runtime manifest. That manifest should include only what the published app needs: scene selection, asset URLs/hashes, script artifact URLs/hashes, module bundle references, and explicit links to existing `playcanvasCanvas` or a future runtime widget. Authoring-only Editor state should not sync to applications by default.
-   Update `publicationSnapshotHash.ts` or reduce PlayCanvas data into existing hashed snapshot fields. If published scene/asset/script artifact content changes, the publication hash must change.
-   If PlayCanvas runtime projection is persisted outside existing `_app_*` tables, add an application sync persistence/diff path comparable to existing modules/packages/layouts sync. Otherwise PLAN must explicitly state which existing persisted fields carry the projection.
-   Serve project assets and generated artifacts through a project-asset route, not the package artifact route. The route contract should include authenticated metahub/project access, immutable cache for hashed generated artifacts, mutable/no-cache behavior for authoring reads, MIME validation, and no absolute path leakage.
-   Runtime `apps-template-mui` should continue to own canvas lifecycle and UI controls. Future Editor-derived runtime projection should feed that runtime through a normalized manifest, not import Editor PCUI/Observer code.
-   UI planning must include localized EN/RU status and validation for missing assets, stale artifacts, checksum conflicts, unsupported asset types, and storage-provider errors without exposing raw absolute paths, raw JSON, DB ids, or internal stack/protocol errors.
-   Future authoring UI must also hide PlayCanvas internal ids unless they are intentionally developer diagnostics. Normal user surfaces should display project names, scene names, asset virtual paths, filenames, and localized recovery/status messages rather than raw UUIDs, Observer JSON, parse pipeline payloads, or filesystem paths.

## Recommended Decision

Proceed to PLAN with a conservative hybrid storage model:

1. Treat PlayCanvas project storage as a metahub authoring store, not package config and not application runtime sync.
2. Use a first-class, versioned project envelope plus a PlayCanvas-specific file namespace for scenes/assets/generated artifacts.
3. Keep TypeScript/TSX source ownership in file-backed Modules and represent PlayCanvas script assets as references plus generated JS/ESM artifact metadata.
4. Publish only a normalized runtime manifest/projection and compiled module bundles; keep Editor-only project state out of `_app_packages` and application sync unless explicitly projected.
5. Extend snapshot export/import/copy/hash behavior in the same implementation slice as the storage model tables/contracts.
6. Defer S3/admin provider configuration, Editor bridge/postMessage/backend emulation, Colyseus authoring, and AI/MCP skills to later briefs/plans.

## Open Questions Before PLAN

-   Should the first implementation use metahub branch system tables, platform-level tables, file-backed JSON envelopes, or a hybrid with DB metadata plus file payloads?
-   If metahub branch system tables are used, what structure version bump and migration path should create them in existing branches?
-   What is the exact PlayCanvas project snapshot section name and versioning strategy?
-   Which asset types are supported first: script artifacts and scene JSON only, or also textures/models/audio?
-   Should generated `.js`/`.mjs` artifacts be stored in snapshots, regenerated on import/publish, or both with checksum validation?
-   What stable id strategy should map Editor asset ids to remappable Universo ids during copy/import?
-   Should package attachment config point to a default PlayCanvas project, or should a metahub allow multiple PlayCanvas projects attached to one Editor package?
-   Should the first runtime projection target the existing `playcanvasCanvas` widget, introduce a new widget config schema, or only define the manifest without changing runtime widgets?
-   Does the first runtime projection need a new application sync persistence table/path, or can it be represented in existing layouts/modules for the first slice?
-   What provider-neutral fields are required now so future S3/admin storage configuration can be added without another project-data migration?
