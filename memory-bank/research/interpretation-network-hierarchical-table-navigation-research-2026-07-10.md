## Key Findings

-   Fact: The screenshots show more than two images. The inspected set contains twelve PNG files from the requested time window. Together they show the target workflow: legacy tree view, switch to table, progressive descent, breadcrumb path chips, ellipsis for hidden prefix, upward breadcrumb clicks, left-side hierarchy-derived row labels, colored child cells, right-side material/content pane, and an optional vertical toolbar.
-   Fact: The screenshots also show raw diagnostic blocks for file addresses, child-cell addresses, and tree addresses. These are historical diagnostics, not acceptable normal runtime UI for the new platform.
-   Fact: Key screenshot-to-requirement anchors are:

| Screenshot                      | Observation                                                                                                                       | Planning implication                                                                                     |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| `image_2026-07-10_15-49-29.png` | Legacy tree view, table-toggle affordance, right content pane, raw file/child/tree address diagnostics.                           | Keep diagnostics out of normal runtime UI; preserve visible content-pane sync and view switching.        |
| `image_2026-07-10_15-52-55.png` | Table projection after switching: breadcrumb chips, focused title, hierarchy-derived row labels, colored child cells, right pane. | Default Table must derive rows from hierarchy, not from separately authored Row/Column entities.         |

-   Fact: The historical Elm implementation maintains navigation through path update helpers. `pathUpdate` trims a path to the clicked node, while `pathStackUpdate` prepends a newly visited node unless it already exists in the current path. This supports the requested behavior model, but the Elm code and product-specific names should not be ported.
-   Fact: The public lmn.rs pages are table-first domain references: the homepage exposes "Krugozor" as nested colored HTML tables, and the numbered table pages show many table families and colored grid cells. This supports table-based domain representation, but not any direct code dependency.
-   Fact: MUI Breadcrumbs currently supports the main truncation mechanics needed for the feature: `maxItems`, `itemsBeforeCollapse`, `itemsAfterCollapse`, `expandText`, custom separators, slots, and an `aria-label`.
-   Fact: The most useful current MUI evidence came from the official docs-site Context7 id `/websites/mui_material-ui` and the public docs pages. A separate query against `/mui/material-ui/v7_3_2` was less relevant for Breadcrumbs; PLAN should rely on official docs-site Breadcrumbs documentation plus the package's Material UI v7 README context instead of overclaiming a version-specific snippet.
-   Fact: WAI-ARIA APG treats breadcrumbs as a navigation landmark containing parent links in hierarchical order. If the current item is a link, `aria-current="page"` is recommended; otherwise it is optional.
-   Fact: Playwright docs recommend web-first assertions and resilient locators. The downstream E2E should assert visible, user-facing breadcrumb/table behavior with `getByRole`, `getByText`, `getByTestId`, `toBeVisible`, `toHaveText`, and viewport/no-overflow helpers instead of CSS-chain assertions.
-   Fact: Repository Playwright E2E should use wrapper-owned startup and the E2E target `http://127.0.0.1:3100`, not `pnpm dev`. Before E2E wrappers, PLAN should account for the repository Node 22 requirement in non-interactive shells. Local Supabase minimal is only needed when hosted E2E is unavailable or the local-Supabase workflow itself is under test.
-   Fact: The current Interpretation Network template and fixture already seed `matrixMode: "hierarchicalCells"`, `allowedMatrixViews: ["table", "horizontalRows", "verticalTree"]`, `defaultMatrixView: "table"`, `hierarchyRowMode: "focusedPath"`, and `allowNewAxesInCellDialog: false`.
-   Fact: The runtime fallback in `apps-template-mui` still defaults to `allowedMatrixViews: ["horizontalRows"]` and `defaultMatrixView: "horizontalRows"` when config is missing. That fallback no longer matches the desired Interpretation Network product default.
-   Fact: The current `buildMatrixTableModel(cells)` builds a rectangular matrix from persisted `rowKey` and `colKey` values. It does not derive rows, breadcrumbs, or the visible level from `parentCellId`, focused path, depth, and child order.
-   Fact: The current table renderer includes row/column headers and Add row/Add column controls for the independent-axis scenario. That behavior must remain configurable as a secondary table variant, but it must not be the default Interpretation Network table projection.
-   Fact: `selectedCellId` currently drives details/material pane state and `toFocusedMatrixHierarchyRows`. Breadcrumb navigation must update selection coherently, not only a separate local breadcrumb state.
-   Fact: Route state currently persists structure id only through `useStructureRoute`; `openStructure` resets `selectedCellId` to `null`. The brief requires root auto-focus when exactly one root cell exists, otherwise a localized empty/repair state. Reload-stable focused-cell navigation would require an explicit route/state extension.
-   Fact: Table DnD and empty-slot movement are coupled to row/column slot IDs, and `buildTableSlotMoveUpdates` has a hierarchical-mode path that omits `ParentCellId`. PLAN must explicitly decide whether hierarchical Table disables table-slot DnD, remaps it to parent/child moves, or preserves the current table-slot semantics with updated tests.
-   Fact: The Application settings allowlist, equality check, parsing, and strict Zod schema must be extended for any new settings. Otherwise new `breadcrumbDepth`, `toolbarLayout`, or `tableProjection` keys will be dropped or rejected. `ApplicationSettings.test.tsx` already has save-to-all-active-widgets and divergent-widget normalization scenarios that should be extended, not bypassed.
-   Fact: Existing E2E helpers currently protect old table behavior. In particular, default table runtime assertions expect row/column headers and Add row/Add column behavior rather than screenshot-like hierarchical breadcrumbs.

## Conflicts And Uncertainty

-   The brief metadata says `Affected Projects: n/a`, but the implementation clearly spans `apps-template-mui`, `applications-frontend`, `universo-react-types`, `metahubs-backend`, fixture tooling, and E2E tests.
-   The brief says hierarchical Table is the default. Template/fixture already set `defaultMatrixView: "table"`, but the runtime table model and runtime fallback still do not implement the requested hierarchical table semantics. PLAN must treat this as a behavior/model change, not only a default change.
-   The latest user clarification resolves the toolbar ambiguity: the vertical toolbar must be planned as a configuration option, while the current horizontal toolbar remains the default. Any older wording that leaves toolbar layout "only if PLAN keeps this in scope" should be superseded by this decision.
-   The exact configuration shape is still open. Candidate approaches are either deriving table projection from `matrixMode` (`hierarchicalCells + table` means hierarchical breadcrumb projection; `independentRows + table` means independent axes), or adding an explicit generic setting such as `tableProjection: "hierarchicalPath" | "independentAxes"`. The first option is simpler, but PLAN must confirm it can support future mixed needs without hidden behavior.
-   The exact finite breadcrumb depth semantics need implementation precision. The user request says a finite value such as `4` should show the last four path items with an ellipsis before them. MUI defaults show both a leading and trailing subset when collapsed; if the desired UI is "ellipsis plus last N only", PLAN may need `itemsBeforeCollapse=0` if supported visually, or a small local accessible wrapper/menu while preserving MUI style.
-   URL/runtime state currently persists selected structure, not selected cell. The remaining product decision is whether breadcrumb focus must survive reload/back-forward navigation; if yes, PLAN must add a bounded route/state extension instead of assuming it already exists.
-   Root ambiguity is not solved by selecting the first root silently. The current state code can derive roots, but the requirement is exactly-one-root auto-focus; no-root and multi-root cases should be localized repair/ambiguous states or an explicit PLAN decision.

## Project Implications

-   Add a generic hierarchical table view-model in `apps-template-mui` for `hierarchicalCells + table`. It should derive:
    -   full ancestor path for breadcrumbs;
    -   display path after configured truncation;
    -   hidden-prefix expansion metadata;
    -   focused cell title;
    -   row labels from the current/focused hierarchical level;
    -   visible child cells from the selected/focused path;
    -   user-facing labels and optional position labels only.
-   Keep the independent row/column table behavior available as a secondary configuration path. This is a product contract, not a requirement to preserve current code internals. Refactoring `buildMatrixTableModel`, table DnD, and table renderer is acceptable.
-   Make root/no-selection behavior explicit. Opening a structure with exactly one root should focus that root automatically. No-root or ambiguous-root cases should use localized recovery states and should not expose hidden IDs.
-   Update the runtime config fallback. A published Interpretation Network workspace without explicit view config should not silently fall back to horizontal rows if the product default is table.
-   Extend the shared strict schema in `@universo-react/types` before adding new settings:
    -   breadcrumb depth: default full path represented consistently as absence, `null`, or an explicit `"full"` value; finite positive integer choices such as 4 are normalized/rejected when invalid;
    -   toolbar layout: default `horizontal`, optional `vertical`;
    -   optionally table projection if PLAN chooses an explicit field.
-   Extend Application settings parsing, save allowlist, equality checks, settings UI, localization, and tests so the new settings round-trip through materialized `interpretationNetworkWorkspace` widgets. If Table view is disabled or not allowed, dependent breadcrumb-depth/table defaults should be disabled or normalized with a user-facing explanation.
-   Update the metahub template and product fixture contract only through the generator/drift workflow. Do not hand-edit the committed fixture as the source of truth. PLAN should explicitly cover `interpretationNetworkTemplateShape.test.ts`, `metahubs-interpretation-network-app-export.spec.ts`, `interpretationNetworkFixtureContract.ts`, `checkInterpretationNetworkFixtureContract.ts`, `checkInterpretationNetworkFixtureDrift.ts`, and `tools/fixtures/metahubs-interpretation-network-app-snapshot.json`.
-   Replace or split old E2E assertions so they do not lock the default runtime to independent row/column table behavior. The flow spec should assert hierarchical breadcrumb Table by default and keep independent-axis Table assertions only under the secondary mode. The visual spec should be extended because it already provides screenshot attachment and no-page-horizontal-overflow evidence.
-   Add feature-specific no-leakage checks for historical raw diagnostics: file addresses, child-cell addresses, tree addresses, UUID-only labels, CID-like hashes, raw JSON, `[object Object]`, and internal field names.
-   Preserve `apps-template-mui` visual language. Use existing MUI dashboard primitives, compare against `NavbarBreadcrumbs.tsx` and `PageContainer.tsx` before adding a Matrix-specific path/header, use MUI Table/Breadcrumbs/Tooltip/IconButton/Button/Stack patterns, localized labels, constrained table scroll, and viewport matrix proof. Do not introduce a configuration-specific shell.
-   Preserve `apps-template-mui` isolation: no imports from legacy `@universo-react/template-mui` or legacy feature packages. Component duplication across the isolation boundary is acceptable; shared extraction should go only to stable non-legacy packages with explicit justification.
-   Keep Runtime UX Gate checks in scope even when they are not the main feature: semantic long-text fields remain multiline, optional resource-source fields do not show premature errors before user input, validation messages are localized/user-facing, keyboard paths use accessible names, and page-level horizontal overflow remains forbidden.
-   Do not introduce product-specific code names. Implementation terms should stay generic: Interpretation Network, Matrix, Table, Structure, Material, breadcrumb/path, hierarchical projection, independent axes.

## Recommended Decision

Use the existing peer view model (`table`, `horizontalRows`, `verticalTree`) and implement hierarchical Table as the primary `table` projection when `matrixMode === "hierarchicalCells"`. Keep independent-axis Table as the `table` projection for `matrixMode === "independentRows"` or behind an explicit generic projection setting if PLAN finds that deriving it from `matrixMode` is too implicit.

Add two new generic settings to the required PLAN scope:

-   `breadcrumbDepth`: full path by default; finite values show the last N path items with an accessible ellipsis/hidden-prefix control.
-   `toolbarLayout`: `horizontal` by default; `vertical` is opt-in and must reuse the same actions and MUI styling.

Implement breadcrumb clicks as real focus/selection navigation. A breadcrumb click must update focused title, table rows, visible child cells, selected cell, and material/details pane. If PLAN requires reload-stable matrix focus, add explicit route/state support because current routing stores only the selected structure.

Treat the old row/column table implementation as refactorable. The required compatibility is behavioral availability as a secondary setting, not preserving the current function/component shape.

## Open Questions Before PLAN

-   Should the table projection be derived solely from `matrixMode`, or should a new explicit setting such as `tableProjection` be added for clarity?
-   Should finite breadcrumb depth be a free positive integer, a constrained select with common values, or both UI select plus schema integer?
-   When a path is truncated, should the ellipsis expand inline to the hidden prefix, open a menu, or use MUI's default collapsed Breadcrumbs behavior?
-   Should focused cell/breadcrumb context persist in URL/back-forward/reload state, or is session-only focus sufficient? Current route state only includes selected structure.
-   How should hierarchical table handle DnD and empty-slot actions: disabled in hierarchical projection, remapped to parent/child moves, or redesigned as explicit actions?
