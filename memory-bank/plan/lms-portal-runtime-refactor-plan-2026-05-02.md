## Overview

The current LMS snapshot and runtime demonstrate platform plumbing, but the generated application does not yet feel like a coherent LMS. The default published app still exposes demo-like widgets and a flat internal metadata menu:

1. The main area shows a module widget that behaves like static demo content.
2. The right drawer shows statistics and QR widgets globally.
3. The left menu expands low-level catalogs instead of exposing a concise user portal.
4. Several MUI template demo leftovers can still leak into runtime surfaces.

The next implementation should turn the LMS fixture into a product-quality configuration generated by the platform:

1. The metahub defines metadata types, entities, scripts, default layouts, and default data.
2. The application control panel defines tenant-level navigation, branding, enabled modules, report presets, and application-wide runtime settings.
3. The published application renders a clean learner/admin LMS portal from generic runtime capabilities.
4. Each workspace can override local learning settings, visibility, knowledge spaces, mentors, supervisors, and workspace-specific content.

The result must be a regenerated product Playwright fixture at:

- [tools/fixtures/metahubs-lms-app-snapshot.json](../../tools/fixtures/metahubs-lms-app-snapshot.json)

This fixture must be importable on a clean database, create a metahub, create an application, and immediately show a working LMS-like portal without post-import manual fixes.

QA constraint:

- The implementation must first extend existing runtime widgets, editors, and template components. New UI components or new widget keys are allowed only after a short code-level proof shows that the existing `menuWidget`, `columnsContainer`, `overviewCards`, `detailsTable`, `ItemCard`, `FlowListTable`, `ViewHeaderMUI`, `ToolbarControls`, and `RuntimeWorkspacesPage` patterns cannot support the needed behavior cleanly.

## Validated Findings

### F-1: The LMS template seeds the reported global widgets

[packages/metahubs-backend/base/src/domains/templates/data/lms.template.ts](../../packages/metahubs-backend/base/src/domains/templates/data/lms.template.ts) currently seeds:

- `moduleViewerWidget` in the center zone.
- `statsViewerWidget` in the right zone.
- `qrCodeWidget` in the right zone.
- `menuWidget` with `autoShowAllCatalogs: true`.

This directly explains the global module, statistics, and QR surfaces in the published application.

### F-2: The committed LMS fixture preserves the same runtime shape

[tools/fixtures/metahubs-lms-app-snapshot.json](../../tools/fixtures/metahubs-lms-app-snapshot.json) currently contains one layout with eight widgets:

- center: `detailsTitle`, `moduleViewerWidget`, `detailsTable`.
- left: `menuWidget`.
- right: `statsViewerWidget`, `qrCodeWidget`.
- top: `appNavbar`, `header`.

The fixture also contains the `lms-module-viewer` and `lms-stats-viewer` scripts, which makes the current default LMS portal script-widget centric instead of entity-view centric.

### F-3: The flat sidebar is produced by backend runtime expansion

[packages/applications-backend/base/src/controllers/runtimeRowsController.ts](../../packages/applications-backend/base/src/controllers/runtimeRowsController.ts) expands `autoShowAllCatalogs` into linked runtime collections. This is useful for generic CRUD inspection, but it is not a good default for an LMS learner portal because it exposes implementation catalogs as first-class navigation.

### F-4: The frontend runtime still contains template demo fallbacks

The CREATIVE review confirmed demo leftovers in:

- [packages/apps-template-mui/src/dashboard/components/MainGrid.tsx](../../packages/apps-template-mui/src/dashboard/components/MainGrid.tsx) with static overview cards/charts.
- [packages/apps-template-mui/src/dashboard/components/CardAlert.tsx](../../packages/apps-template-mui/src/dashboard/components/CardAlert.tsx) with demo upgrade copy.
- [packages/apps-template-mui/src/dashboard/components/SelectContent.tsx](../../packages/apps-template-mui/src/dashboard/components/SelectContent.tsx) with demo product names.
- [packages/apps-template-mui/src/dashboard/components/widgetRenderer.tsx](../../packages/apps-template-mui/src/dashboard/components/widgetRenderer.tsx) with demo user profile data and hardcoded widget cases.

These must be converted to data-driven or disabled-by-default runtime surfaces before the LMS fixture can be treated as product-quality.

### F-5: The current fixture contract enforces the wrong product behavior

[tools/testing/e2e/support/lmsFixtureContract.ts](../../tools/testing/e2e/support/lmsFixtureContract.ts) currently requires the same module, statistics, and QR widgets that should be removed from the default LMS landing experience. The contract should become a product contract:

- reject global demo widgets in every standard LMS runtime route and active LMS product layout,
- require concise navigation,
- require real LMS entities and records,
- require localized visible labels,
- require stable scripts and workspace behavior,
- require screenshot-backed runtime proof.

### F-6: Existing docs are now stale for the desired LMS direction

The GitBook LMS docs currently describe the QR, module, and statistics widgets as core LMS behavior. Some LMS pages also reuse a quiz tutorial screenshot instead of a real LMS runtime screenshot. These pages must be refreshed after the runtime rebuild.

## External Product References

The iSpring LMS documentation should be used as a product reference, not as an implementation blueprint.

Important validated patterns:

1. The user portal navigation is configurable by administrators, has a quick access menu, moves remaining modules into an overflow menu, and supports a selected start page. iSpring documents a maximum of six quick access modules.
   - Source: https://ispringhelpdocs.com/ispring-learn/navigation-menu-in-the-user-portal-100871437.html
2. The Knowledge Base is a user portal module with spaces, folders, articles, search, recent items, bookmarks, and permissions.
   - Source: https://ispringhelpdocs.com/ispring-learn/knowledge-base-128352839.html
3. Development Plans cover onboarding, career paths, mentors, supervisors, stages, tasks, and progress.
   - Source: https://ispringhelpdocs.com/ispring-learn/development-plans-128352776.html
4. Course Progress reports use summary metrics plus a table, including completed, in progress, not started, and overdue states.
   - Source: https://ispringhelpdocs.com/ispring-learn/course-progress-22610432.html

Context7 documentation checks confirmed the implementation direction:

1. TanStack Query mutations should invalidate stable query keys in `onSuccess`, and awaited invalidations keep mutation state aligned with refetching.
2. MUI layouts should use the current Grid and component composition patterns instead of custom one-off layout CSS.
3. Playwright tests should rely on locators, web-first assertions, and screenshot assertions for visual proof.

Accepted directions:

1. The current LMS fixture is too widget/demo-driven and must become entity-first.
2. The published runtime must present a concise user portal instead of a raw metadata explorer.
3. The application must keep metahub, application, and workspace responsibilities separate.
4. LMS modules such as course modules, progress, access links, and quiz responses should be nested or contextual, not top-level learner navigation.
5. Scripts should calculate business behavior such as progress, eligibility, due states, and access envelopes.
6. Admin, learner, supervisor, and workspace contexts should be part of the long-term design direction.

Constrained or deferred directions:

1. Do not build a new role-specific frontend shell before the generic menu/layout/runtime primitives are corrected.
2. Do not create a separate widget engine for LMS dashboards in the MVP; upgrade existing runtime surfaces first.
3. Do not create a broad `ContentSpace` or `Project` metadata type before the current workspace and Page preset boundaries are validated.
4. Do not move Page into built-in entity kinds for the MVP.
5. Do not import every enterprise LMS feature into the first implementation pass; certificates, SSO, notification automation, OJT, 360 review, and advanced reporting should follow after the fixture proves the core portal.
6. Treat external LMS comparisons as product guidance, not as a mandate to clone screens or introduce platform-specific hardcoding.

## Goals and Non-Goals

### Goals

1. Remove global LMS module, statistics, and QR widgets from the default published LMS application.
2. Replace `autoShowAllCatalogs` with explicit, workspace-aware portal navigation and a safe extension path for role-aware navigation.
3. Keep runtime improvements generic so they can support non-LMS metahubs later.
4. Use the original MUI dashboard visual language from [packages/apps-template-mui](../../packages/apps-template-mui) and [.backup/templates/dashboard](../../.backup/templates/dashboard).
5. Build the LMS configuration from entity-first metadata objects, scripts, layouts, and workspace settings.
6. Add a lightweight Page metadata type or preset when it is useful for block-structured content.
7. Regenerate [tools/fixtures/metahubs-lms-app-snapshot.json](../../tools/fixtures/metahubs-lms-app-snapshot.json) through the product Playwright generator.
8. Add deep Jest, Vitest, and Playwright coverage, including real screenshots.
9. Update package READMEs and GitBook docs in [docs](../../docs).
10. Keep all visible text internationalized from the start.
11. Use UUID v7 consistently for newly generated entities and records.

### Non-Goals

1. Do not build a pixel-perfect clone of iSpring LMS.
2. Do not create an LMS-only frontend shell when generic runtime capabilities can express the same behavior.
3. Do not keep old fixture behavior just to preserve disposable test data.
4. Do not manually edit the committed JSON fixture as the source of truth.
5. Do not implement every enterprise LMS module in the first MVP pass.
6. Do not bump metahub template or schema versions for this rebuild. If implementation uncovers a real compatibility boundary, stop and review the boundary explicitly instead of silently increasing versions.

## Target Architecture

### Platform Layers

1. **Metahub layer**
   - Owns metadata entities, field schemas, default layouts, scripts, template seed data, and publication snapshots.
   - Defines the LMS logical core with Hubs, Catalogs, Sets, Enumerations, and optionally Pages.

2. **Application management layer**
   - Owns tenant-level runtime settings:
     - enabled portal modules,
     - quick access navigation,
     - start page,
     - branding,
     - report presets,
     - workspace availability,
     - future role visibility defaults when role-aware portal navigation is implemented.

3. **Published application runtime**
   - Renders generic entity views, page views, cards, tables, filters, dashboards, and context actions.
   - Uses [packages/apps-template-mui](../../packages/apps-template-mui) as the isolated published-app template.
   - Does not hardcode LMS-only logic unless it is expressed as data, scripts, or metadata.

4. **Workspace layer**
   - Owns local overrides:
     - visible courses,
     - local catalog availability,
     - workspace knowledge spaces,
     - mentors and supervisors,
     - local news,
     - local progress filters,
     - workspace-specific dashboard cards.

### Runtime Capability Additions

1. **Generic menu widget extension**
   - Extend the existing `menuWidget` contract and both existing menu editors.
   - Preserve current broad behavior for templates that need it.
   - Add only the fields required for MVP quick access and overflow first:
     - explicit primary items,
     - primary item limit,
     - overflow label,
     - start page,
     - workspace item placement.
   - Add broader modes such as `boundHub`, `filteredByKind`, and `grouped` only when they are required by a validated runtime case.

2. **Generic portal start page**
   - Store `defaultSection` or `startPage` at layout/menu level.
   - If the configured page is inaccessible, fall back to a safe default.
   - Do not rely on the first menu item as the implicit runtime start section.

3. **Existing columns container upgrade**
   - Extend the current `columnsContainer` instead of creating a parallel content-grid widget.
   - Preserve nested widget IDs, config, responsive spans, and surface props.
   - Support a MUI 12-column layout similar to the original dashboard:
     - primary content at `lg: 9`,
     - secondary inline stack at `lg: 3`,
     - no global right drawer unless a layout explicitly chooses it.

4. **Existing overview cards upgrade**
   - Replace static demo cards with script-backed, query-backed, or runtime-row-backed summary data.
   - Reuse the current `overviewCards` / `StatCard` presentation before adding any new summary widget key.
   - Render using the same MUI card/grid style as the original dashboard.
   - Keep metrics domain-neutral.

5. **Page metadata preset and optional renderer**
   - Implement Page first as a custom entity type preset, not as a new built-in entity kind.
   - Use existing entity components (`dataSchema`, `records`, `layoutConfig`, `runtimeBehavior`, `physicalTable`) and existing runtime row/card/table surfaces first.
   - Add a block renderer only if JSON/TABLE-backed page content cannot be presented through existing runtime components without poor UX.
   - Render block-structured page data from a Page metadata preset when the renderer is justified.
   - Use it for LMS course pages, knowledge articles, news, or static workspace instructions.

6. **Contextual public access**
   - Keep public access links and QR generation, but show them as contextual actions for access-link pages or course sharing.
   - Do not show QR as a global right-side widget on every LMS screen.

## Domain Model Direction

### MVP Metadata Objects

The LMS fixture should start with a stable, evaluable MVP model:

1. **Learning Hub**
   - The root metahub concept for the LMS configuration.

2. **Courses**
   - Primary learning objects.
   - Fields: title, code, summary, cover, status, difficulty, duration, owner, category, visibility, due policy, completion policy.

3. **Course Modules**
   - Child learning units inside courses.
   - Fields: title, content type, order, estimated time, content reference, completion mode.

4. **Pages**
   - Lightweight block-structured content.
   - Used for course lessons, knowledge articles, onboarding instructions, and news.
   - Page must start as a custom entity type preset, not as a new built-in kind.

5. **Quizzes and Quiz Responses**
   - Keep the existing quiz path but make it course/module scoped.

6. **Learners**
   - Represent LMS learner profiles without duplicating platform authentication as the system of record.

7. **Groups and Departments**
   - Used for assignments, visibility, reporting, and workspace membership mapping.

8. **Assignments**
   - Assign courses or learning tracks to learners/groups.
   - Fields: assignee, due date, status, source, workspace, assigned by.

9. **Enrollments and Progress**
   - Track course/module state:
     - not started,
     - in progress,
     - completed,
     - overdue,
     - failed,
     - exempt.

10. **Knowledge Spaces and Knowledge Articles**
    - Entity-backed knowledge base modules with search-friendly fields.

11. **News Posts**
    - Lightweight announcements for the portal and workspaces.

12. **Development Plans**
    - MVP-level plans with stages, mentors, supervisors, and tasks.
    - Full enterprise workflows can follow after the MVP is stable.

13. **Access Links**
    - Contextual public links for guest learning or quiz flows.
    - QR is derived from access links and is not a global dashboard widget.

### Navigation MVP

The default learner portal should expose a concise, product-like navigation:

1. Home
2. Learning
3. Catalog
4. Knowledge
5. Development
6. Team or More

Admin and authoring areas should be separated from learner navigation:

1. Users
2. Learning Content
3. Assignments
4. Reports
5. Settings

Low-level catalogs such as progress records, quiz responses, access links, and internal modules should be reachable through contextual pages, admin sections, or entity detail tabs, not as default learner sidebar items.

   - preserve current behavior for existing generic layouts,
   - keep `autoShowAllCatalogs` available for admin/debug/self-hosted layouts,

6. Add a default section resolver instead of depending on the first menu item in [useCrudDashboard.ts](../../packages/apps-template-mui/src/hooks/useCrudDashboard.ts).

### Phase 3 - Add Generic Runtime View Primitives

1. Upgrade the existing `columnsContainer` model that preserves:
   - nested widget IDs,
   - nested widget configs,
   - responsive column spans,
   - surface props,
   - loading and empty states.
2. Update both existing columns-container editors:
   - [packages/metahubs-frontend/base/src/domains/layouts/ui/ColumnsContainerEditorDialog.tsx](../../packages/metahubs-frontend/base/src/domains/layouts/ui/ColumnsContainerEditorDialog.tsx)

   - no hardcoded LMS copy.
4. Add a Page custom entity type preset proof:
   - register it as an optional `entity_type_preset`,
   - do not add it to `BuiltinEntityKinds`,
   - verify it behaves as a runtime catalog-like target because custom kinds are treated as runtime collections.
5. Add a generic page renderer only after the preset proof:
   - block types: text, media, checklist, embedded entity, callout, table, link,
   - data stored as JSON with a validated schema,
   - all rendered labels are i18n-aware.
6. Change right-zone behavior:
   - `hidden` by default for LMS,
   - `inlineGrid` for dashboard side stacks,
   - `drawer` only for explicit context panels.
7. Remove or gate remaining demo template surfaces:
   - overview cards,
   - demo charts,
   - demo alert copy,
   - demo user profile values,
   - demo product selector values.

### Phase 4 - Rebuild the LMS Seed Model

1. Replace the current themed sample with practical corporate learning content:
   - onboarding,
   - product training,
   - compliance,
   - sales enablement,
   - support playbooks,
   - manager development.
2. Seed bilingual EN/RU labels and records through the template/generator path.
3. Use UUID v7 for generated IDs.
4. Seed realistic relationships:
   - courses to modules,
   - modules to pages/quizzes,
   - learners to groups/departments,
   - assignments to learners/groups,
   - enrollments to progress,
   - development plans to stages/tasks,
   - knowledge articles to spaces/folders.
5. Keep low-level operational entities out of the default learner menu.

### Phase 5 - Application Control Panel Settings

1. Add or refine application settings for:
   - enabled portal modules,
   - quick access order,
   - start page,
   - branding,
   - workspace feature flags,
   - report presets,
   - public access defaults.
2. Extend the existing strict `applicationDialogSettingsSchema` instead of storing arbitrary JSON settings.
3. Update the existing application settings UI in [packages/applications-frontend](../../packages/applications-frontend) using current form/dialog components.
4. Use TanStack Query for mutations and cache invalidation.
5. Await invalidations for runtime/layout/settings mutations so pending state matches refetch lifecycle.
6. Persist settings through SQL-first backend stores with request-scoped executors.
7. Ensure settings are included in sync/materialization from publication to application runtime.

### Phase 6 - Published LMS Runtime

1. Build the default portal views from generic components:
   - Home: assigned learning, due items, recent knowledge, progress summary.
   - Learning: course cards/list, status filters, detail drawer or page.
   - Catalog: available courses and learning tracks.
   - Knowledge: spaces, folders, articles, search.

## Code Quality Examples

### Safe SQL Store Pattern

```ts
import type { DbExecutor } from '@universo/utils'
import { qSchemaTable } from '@universo/database'

interface UpdateCourseStatusInput {
  schema: string
  courseId: string
  status: 'draft' | 'published' | 'archived'
}

export async function updateCourseStatus(
  executor: DbExecutor,
  input: UpdateCourseStatusInput
): Promise<{ id: string; status: string }> {
  const table = qSchemaTable(input.schema, 'lms_courses')

  const rows = await executor.query<{ id: string; status: string }>(
    `
      update ${table}
         set status = $2,
             updated_at = now()
       where id = $1
       returning id, status
    `,
    [input.courseId, input.status]
  )

  const row = rows[0]
  if (!row) {
    throw new Error('Course was not found or cannot be updated')
  }

  return row
}
```

Expected implementation notes:

1. Dynamic identifiers are quoted through shared helpers.
2. Values are passed through `$1`, `$2` parameters.
3. Mutations use `RETURNING`.
4. Zero-row updates fail closed.

const saveNavigationMutation = useMutation({

### Script Contract Pattern

```ts
import { AtServer, ExtensionScript } from '@universo/extension-sdk'

interface ProgressSummaryInput {
  workspaceId?: string | null
  learnerId: string
}

type ProgressSummary = {
  completed: number
  inProgress: number
  overdue: number
}

const emptySummary = (): ProgressSummary => ({
  completed: 0,
  inProgress: 0,
  overdue: 0,
})

const isRecord = (value: unknown): value is Record<string, unknown> =>
  typeof value === 'object' && value !== null

export default class LmsProgressSummaryScript extends ExtensionScript {
  @AtServer()
  async calculateProgressSummary(input?: ProgressSummaryInput): Promise<ProgressSummary> {
    const learnerId = input?.learnerId?.trim()
    if (!learnerId) {
      throw new Error('learnerId is required')
    }

    const rows = await this.ctx.records.list('learning-progress', {
      learnerId,
      workspaceId: input?.workspaceId ?? null,
      limit: 500,
    })

    return rows.reduce((summary, row) => {
      if (!isRecord(row)) {
        return summary
      }

      if (row.status === 'completed') {
        summary.completed += 1
      } else if (row.status === 'overdue') {
        summary.overdue += 1
      } else {
        summary.inProgress += 1
      }

      return summary
    }, emptySummary())
  }
}
```

Attach this script with `records.read`; only client-facing widgets that proxy to a server method should also request `rpc.client`.

Incorrect implementation to avoid:

```ts
import type { RuntimeScriptContext } from '@universo/types'

export async function calculateProgressSummary(ctx: RuntimeScriptContext) {
  // There is no ctx.query() API in the current extension SDK.
  return await ctx.query('lms.progress.summary')
}
```

Expected implementation notes:

1. Scripts validate inputs before reading runtime data.
2. Scripts use the current `@universo/extension-sdk` contract: `ExtensionScript`, decorators, and `this.ctx.records`.
3. Scripts request only the capabilities they need.
4. Scripts return a small serialized model.
5. Scripts do not embed static demo UI copy.

### Existing SDK Contract Guardrail

The implementation must not invent a new runtime script host API for the LMS plan. It should extend the existing SDK only through explicit package-level changes in `packages/extension-sdk`, `packages/scripting-engine`, and runtime script service tests.

Current supported surfaces:

1. `this.ctx.records.list/get/findByCodename/create/update/delete`
2. `this.ctx.metadata.getAttachedEntity/getByCodename`
3. `this.ctx.callServerMethod`
4. Reserved optional APIs such as `http`, `state`, `log`, and `i18n`

Any new script API must include SDK types, compiler coverage, isolated runtime coverage, backend capability checks, client context behavior, and script editor autocomplete updates in the same implementation slice.

## QA Review Refinements

The plan was checked against the current codebase after the first draft. These refinements are mandatory for implementation:

1. **Do not introduce parallel UI primitives too early**
   - `columnsContainer` already exists in shared types, metahub layout authoring, application layout authoring, and runtime rendering.
   - The first change must preserve nested widget config in the existing `columnsContainer` schema and editors.
   - A new `contentGrid` widget is not justified for the MVP.

2. **Do not add a new summary widget before upgrading existing overview cards**
   - `MainGrid` already renders `overviewCards` through `StatCard`.
   - The implementation should make this surface data-driven or disabled-by-default.
   - A new `summaryCardsWidget` key is allowed only if the existing overview surface cannot remain backwards-compatible.

3. **Page must start as a custom entity type preset**
   - Current built-in kinds are `catalog`, `set`, `enumeration`, and `hub`.
   - Custom entity kinds are supported through entity type definitions and presets.
   - Runtime collection logic already treats non-built-in non-hub/set/enumeration kinds as catalog-like runtime targets.
   - Therefore Page should not be added to `BuiltinEntityKinds` in the MVP.

4. **Both layout authoring surfaces must be updated together**
   - Metahub layout editor and application layout editor currently have separate menu and columns-container dialogs.
   - Any config schema extension must update both editors, both i18n namespaces, shared types, validators, and E2E coverage.

5. **Application settings must stay schema-validated**
   - The applications backend currently uses a strict Zod `applicationDialogSettingsSchema`.
   - LMS portal settings should extend that schema or a shared schema derived from it.
   - Do not store unchecked free-form settings in `applications.settings`.

6. **Keep old broad menu behavior where it is valid**
   - `autoShowAllCatalogs` is still used by self-hosted/basic/admin-style layouts.
   - The LMS template must disable it, but the runtime capability should not be removed globally.

7. **Remove widgets from the default LMS experience, not necessarily from the platform**
   - `moduleViewerWidget`, `statsViewerWidget`, and `qrCodeWidget` should disappear from active LMS product layouts and all standard LMS portal routes.
   - QR/public access can remain as contextual functionality.
   - Existing widget keys may remain available until generic replacements cover their use cases.

8. **Tests must be changed where they currently encode the wrong product contract**
   - `lmsFixtureContract.ts`, `snapshot-import-lms-runtime.spec.ts`, `lms-statistics.spec.ts`, and `lms-qr-code.spec.ts` currently expect the old global widgets.
   - These tests must be rewritten or moved to contextual widget coverage.

9. **TanStack Query invalidation should be aligned across packages**
   - Some runtime mutations already await invalidations, while apps-template row mutations currently fire invalidations without awaiting them.
   - New or touched mutations should follow the awaited invalidation pattern verified in Context7.

10. **Scripts must use the existing sandbox and capability model**
    - The scripting engine already uses isolated execution, module-role metadata, capability checks, and public RPC gating.
    - LMS scripts must request only required capabilities and must not bypass runtime services with direct network or database access.
    - Add tests for denied capabilities, invalid script inputs, and script error states where LMS runtime depends on scripts.

11. **Playwright proof must include negative assertions**
    - The product E2E flow must assert absence of:
      - `Orbital navigation`,
      - `Orbital Academy statistics`,
      - `Module access QR`,
      - default right drawer on standard LMS portal routes.

12. **Documentation cleanup must remove stale LMS claims**
    - LMS GitBook pages currently still describe Orbital Academy, global QR/module/statistics widgets, and reuse `quiz-tutorial/runtime-quiz.png`.
    - The docs update must replace those references with real LMS portal screenshots and the new corporate learning fixture.
    - Generic platform pages may keep quiz screenshots only when they are explicitly documenting the quiz tutorial, not LMS runtime.

## Risks and Mitigations

### R-1: Overbuilding a full enterprise LMS before the MVP is stable

Mitigation:

- Implement the learner/admin portal MVP first.
- Keep development plans, reports, and knowledge base at a useful but limited depth.
- Leave advanced workflows for follow-up plans.

### R-2: LMS-specific logic leaking into generic runtime packages

Mitigation:

- Add generic runtime capabilities first.
- Express LMS behavior through template seed, layout config, entities, and scripts.
- Add tests that verify non-LMS layouts still work.

### R-3: Menu settings drifting between metahub, application sync, and runtime

Mitigation:

- Use shared types in [packages/universo-types](../../packages/universo-types).
- Add backend sync tests and frontend rendering tests.
- Add fixture contract checks for the generated snapshot.

### R-4: UI parity degrading as new runtime views are added

Mitigation:

- Reuse [apps-template-mui](../../packages/apps-template-mui) and [universo-template-mui](../../packages/universo-template-mui) components.
- Compare against [.backup/templates/dashboard](../../.backup/templates/dashboard) where relevant.
- Add Playwright screenshots for desktop and mobile.

### R-5: Demo content becoming static and misleading again

Mitigation:

- Store content as seeded entities and runtime rows.
- Use scripts only for computed logic.
- Test that visible widgets reflect seeded data rather than embedded strings.

### R-6: i18n gaps

Mitigation:

- Add all shared keys to [packages/universo-i18n](../../packages/universo-i18n) when reused.
- Keep package-local keys only for package-local UI.
- Add raw-key tests for LMS runtime surfaces.

### R-7: Fixture JSON becoming hand-maintained

Mitigation:

- Keep the Playwright generator as the source of truth.
- Validate the generated JSON with [lmsFixtureContract.ts](../../tools/testing/e2e/support/lmsFixtureContract.ts).
- Do not edit the fixture manually except as an emergency diagnostic step that is reverted before completion.
