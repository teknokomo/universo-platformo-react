# iSpring-like LMS Platform Roadmap

Date: 2026-05-11  
Mode: PLAN  
Status: Implemented and QA-remediated on 2026-05-12  
Complexity: Level 4, platform architecture and product fixture

## Goal

Design the next platform roadmap that turns the current entity-first LMS configuration into a working iSpring-like LMS application while preserving the Universo architecture:

- Metadata and base logic are defined in the Metahub.
- Application control panel stores tenant-level defaults and final app-wide settings.
- Published application workspaces store local settings, records, and operational data.
- LMS behavior is expressed through generic metadata types, scripts, layouts, widgets, reports, and workspaces instead of LMS-specific runtime forks.
- The final product artifact is regenerated by Playwright into `tools/fixtures/metahubs-lms-app-snapshot.json`.

This plan changes the current LMS template, Playwright generator, fixture contract, and platform primitives in place. It must not bump the Metahub schema version or LMS template version unless a separate compatibility decision explicitly requires it. Legacy demo data does not need to be preserved, and the test database can be recreated.

### QA Review Addendum

The QA review confirmed that the overall architecture is correct, but the implementation plan must be stricter in several areas:

- Do not introduce a new LMS role system. Extend the current application and workspace permission model first.
- Do not introduce a new report widget or report table in V1. Store report definitions as ordinary configured records and execute them through existing runtime data-source descriptors.
- Treat iSpring-like completion statuses as product-critical behavior, not visual labels.
- Cover attendance and certificate lifecycle states in the same normalized status model.
- Keep core runtime record commands limited to `post`, `unpost`, and `void`; model LMS-like actions as configured workflow/lifecycle transitions unless a generic workflow contract is explicitly required.
- Make the existing layout/widget editors better with metadata-backed pickers instead of adding parallel UI.
- Remove or data-drive remaining demo dashboard surfaces before they can appear in a published app.
- Resolve report runner table and column identifiers only from validated published metadata, never from raw API payload identifiers.
- Keep all generated identifiers on the existing UUID v7 path.
- Keep the LMS template and snapshot generator on the current version path unless a separate migration plan is approved.

## Main Gaps

1. Content resources are not yet a first-class generic platform capability.
   The current LMS has Pages and Modules, but an iSpring-like system needs reusable resource metadata for SCORM-like packages, videos, audio, documents, links, embedded content, and authored pages.

2. Learning tracks need enforceable progression rules.
   The current template can seed tracks and module progress, but platform-level generic sequencing and script hooks should enforce order, prerequisites, due dates, availability, attempts, and completion policy.

3. Application and workspace role policies need a generic editor and runtime enforcement surface.
   LMS roles must be configuration labels over the existing application/workspace membership and permission model. Runtime code should enforce generic capabilities such as read, author, assign, grade, report, approve, manage members, and manage settings.

4. Reports are still too close to dashboard widgets.
   The platform needs reusable report definitions, a safe report runner, saved filters, projections, and export preparation. The first implementation must reuse `detailsTable`, charts, `overviewCards`, `columnsContainer`, and existing data-source contracts.

5. Assignments, training events, certificates, and notifications need complete operational workflows.
   The current fixture has several entities and ledgers, but the runtime must support real record commands, status transitions, posting, and visibility rules.

6. Published app UI needs original MUI dashboard parity.
   `packages/apps-template-mui` should keep the original dashboard visual language from `.backup/templates/dashboard`. Workspace cards and any remaining demo copy such as Sitemark or Product tree must be removed or made generic and translated.

7. Fixture seed data is still too small for product evaluation.
   The generated snapshot must represent a credible LMS scenario, not only schema shape.

8. Documentation still has stale screenshots.
   LMS docs should use real screenshots captured from the generated application.

9. Completion statuses need a normalized product model.
   iSpring-like flows depend on statuses such as Not started, In progress, Completed, Failed, Passed, Pending Review, Declined, Accepted, Overdue, and Expired. These states should be generic enumeration/configuration data with lifecycle and ledger meaning.

10. Existing widget editors still expose raw identifiers.
    Application layout editors should use metadata-backed pickers for sections, ledgers, projections, fields, and reports before adding new UI surfaces.

## Architecture Decisions

1. Do not add an LMS runtime module unless a generic platform primitive cannot solve the case.
2. Do not add a separate Document entity type.
   Extend Catalog behavior through `recordBehavior`, table parts, lifecycle, posting, and validation.
3. Keep ledgers as Catalog-backed configurations through `config.ledger`.
4. Keep authored content as Page entities and resource catalogs.
5. Add generic resource, sequence, report, role, and notification primitives where needed.
6. Keep scripts capability-scoped and deterministic.
7. Use `DbExecutor.query()` and schema-qualified parameterized SQL for backend data access.
8. Keep all user-visible text internationalized through package i18n and `packages/universo-i18n`.
9. Keep package boundaries strict: shared contracts in `packages/universo-types`, shared utilities in `packages/universo-utils`, shared UI in `packages/universo-template-mui`, published runtime in `packages/apps-template-mui`.
10. Regenerate `tools/fixtures/metahubs-lms-app-snapshot.json` only through Playwright.
11. Do not bump the Metahub schema version or LMS template version for this work.
12. Extend the existing `owner/admin/editor/member` application role model and workspace roles with configurable capabilities before considering any new role table or subsystem.
13. In V1, reports are Catalog-backed configuration records executed through `runtimeDatasourceDescriptorSchema`; do not add a new report widget key unless existing generic widgets cannot represent the result.
14. New generated ids must use UUID v7 through the existing project helpers or database defaults.
15. Published app dashboard demo surfaces must be either data-driven by runtime configuration or blocked from production layouts.

export default defineLifecycleScript({
  async beforePost(ctx) {
    const progress = await ctx.records.get(ctx.record.id)
    const policy = await ctx.metadata.getConfig('LearningTracks', 'sequencePolicy')

    if (policy.mode === 'sequential') {

        trackId: progress.trackId,
        stepOrder: progress.stepOrder,

          ledger: 'ProgressLedger',
          dimensions: {
            learnerId: progress.learnerId,
            moduleId: progress.moduleId
          },
          measures: {
            completionPercent: progress.completionPercent
          }
        }
      ]
    }
  }
})

Safe SQL example:

The runner/store must resolve table and column identifiers from validated published metadata. API payloads may reference codenames, but must never supply raw table or column names.

```ts
export async function updateReportDefinitionRecord(
  executor: DbExecutor,
  schema: string,
  tableName: string,
  columns: { config: string },
  input: UpdateReportDefinitionInput
): Promise<{ id: string }> {
  const table = qSchemaTable(schema, tableName)
  const configColumn = qColumn(columns.config)
  const result = await executor.query<{ id: string }>(
    `UPDATE ${table}
       SET ${configColumn} = $2::jsonb,
           _upl_updated_at = now()
     WHERE id = $1
       AND _upl_deleted = false
       AND _app_deleted = false
     RETURNING id`,
    [input.id, input.config]
  )

  if (result.rows.length !== 1) {
    throw new MetahubDomainError('REPORT_DEFINITION_NOT_UPDATED', 'Report definition was not updated')
  }

  return result.rows[0]
}
```

## Safety and Performance Requirements

1. All dynamic identifiers in SQL must go through `qSchema`, `qTable`, `qSchemaTable`, or `qColumn`.
2. All SQL values must use `$1`, `$2`, and parameter arrays.
3. Mutations that require confirmation must use `RETURNING` and fail closed on zero rows.
4. Scripts must run with explicit capabilities and no direct database or network access.
5. Runtime lists and reports must use server-side pagination, sorting, and filtering.
6. Widgets must use stable query keys and avoid unbounded fetches.
7. Workspace and role filters must be enforced in backend routes, not only in UI.
8. UI must preserve MUI dashboard responsiveness and avoid text overlap.
9. No visible user text may bypass i18n.
10. Playwright screenshots must be reviewed before accepting UI changes.
11. Workflow actions must not expand the core `post`/`unpost`/`void` command union unless a generic, non-LMS workflow contract explicitly requires it.
12. New ids must use UUID v7 through project helpers or database defaults.
13. Existing UI surfaces must be extended before creating new LMS-specific components.
14. Existing application/workspace permission routes must remain the enforcement boundary.

## Risks

| Risk | Mitigation |
| --- | --- |
| LMS hardcoding returns | Contract checks for generic widgets, generic data sources, and no LMS-only runtime modules |
| Fixture looks real but workflows do not work | Product E2E journeys with posting and screenshots |
| Reports become unsafe SQL | Validate descriptors and generate SQL only from whitelisted fields |
| Role policy leaks workspace data | Backend authorization tests and workspace isolation Playwright tests |
| UI drifts from original MUI dashboard | Screenshot comparisons against the dashboard visual baseline |
| Scripts become too powerful | Capability-scoped script API and negative tests |
| Snapshot is manually edited | Generator-only snapshot updates and fixture contract validation |
| Docs become stale | Link docs updates to the fixture regeneration acceptance checklist |
| New UI fragments diverge from existing surfaces | Extend existing MUI template components and layout editors first |
| Statuses become decorative only | Require status transition tests and ledger/report proof |
| Version bump creates unnecessary migration work | Contract test that the LMS path stays on the current schema/template version |
