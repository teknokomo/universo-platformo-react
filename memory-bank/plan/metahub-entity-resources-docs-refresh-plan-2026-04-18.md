# Metahub Entity Resources, Templates, Fixtures, and Docs Refresh Plan

## Overview

This plan covers the next corrective implementation wave for the entity-first metahub architecture. The target is a stable MVP without preserving legacy behavior: resource tabs must become truly entity-driven, built-in templates must seed the correct resource semantics, snapshot fixtures must be regenerated from the current Playwright product flows, a new empty metahub template must be introduced, and GitBook documentation in both English and Russian must be rewritten against the actual shipped UI.

The plan assumes that the old hardcoded Hubs/Catalogs/Sets/Enumerations implementation is already removed and that further compatibility-only code should be deleted instead of extended. No schema version bump and no metahub template version bump should be introduced in this wave.

## Verified Current State

- `SharedResourcesPage.tsx` already derives visible resource tabs from entity-type component manifests, but the tab labels and surrounding semantics are still tied to old resource naming.
- Built-in metahub templates currently include only `basic` and `basic-demo`.
- Built-in entity presets still define standard kinds through static seed definitions in `packages/metahubs-backend/base/src/domains/templates/data/`.
- `EntityTypeUIConfig` already exists as the shared UI contract for entity types and is validated in backend controllers, backend services, and template manifest validation.
- `EntitiesWorkspace` already owns the create/edit flow for entity types, so resource-tab settings should be added there instead of introducing a second unrelated settings surface.
- The E2E inventory already contains product flows, generators, and docs screenshot generators, so the right path is to modernize and consolidate them, not to invent a second test stack.
- Russian documentation still contains mixed English terminology such as `field definitions`, `fixed values`, and `option values`, which means the docs are not aligned with the current product language.
- The workspace uses the UP-test Supabase project for E2E and Playwright-driven validation on port `3100`.

## Constraints

- Do not preserve legacy route families, labels, adapters, or compatibility shims unless a live runtime dependency proves they are still required.
- Keep all new UI text fully internationalized from the start.
- Keep database access SQL-first and route/service/store layered on the backend.
- Use Playwright product flows and screenshots to verify the real UI instead of documenting inferred behavior.
- Prefer reusable shared types and helpers in `@universo/types` and `@universo/utils` when the same contract is needed across backend, frontend, generators, and docs tooling.
- Use UUID v7 where new identifiers are generated by application code.

## Phase 1: Make Resource Tabs Truly Entity-Driven

- [ ] Extend the existing `EntityTypeUIConfig` contract in `@universo/types` with resource-surface metadata for entity-owned and metahub-level resource tabs.
- [ ] Move resource-tab naming and enablement logic out of hardcoded frontend label maps into entity-type configuration data.
- [ ] Ensure standard presets seed the resource-surface descriptors for:
  `catalog` → `Attributes`
  `set` → `Constants`
  `enumeration` → `Values`
- [ ] Keep the metahub-level Resources page generic:
  it should render only the tabs enabled by current entity types and read the display label from the configured resource surface metadata, not from hardcoded translation keys alone.
- [ ] Ensure future custom entity types can define their own resource tabs through the same contract without editing page-level React code.
- [ ] Reuse the current capability model:
  `ComponentManifest` continues to answer whether a surface is available, while `EntityTypeUIConfig` answers how that surface is presented.

### Design Notes

- The cleanest shape is usually a typed descriptor array rather than scattered booleans:
  one object can carry `resourceKey`, i18n key, fallback label, owning component, route segment, and enablement.
- Avoid letting frontend JSX infer business meaning from component names like `dataSchema` or `optionValues` alone. Those capabilities should gate availability, while the display contract should come from entity configuration.
- Do not introduce a second independent resource-tab registry if the same result can be achieved by extending the existing `EntityTypeUIConfig` plus the current validator and preset-manifest pipeline.

### Example Safe Contract

```ts
export interface EntityResourceSurfaceDefinition {
  key: 'fieldDefinitions' | 'fixedValues' | 'optionValues' | string
  routeSegment: string
  titleKey: string
  fallbackTitle: string
  capability: keyof ComponentManifest
}

export interface EntityTypeUIConfig {
  iconName?: string
  tabs: string[]
  resourceSurfaces?: EntityResourceSurfaceDefinition[]
}
```

## Phase 2: Refactor Frontend Resource Rendering

- [ ] Replace remaining static tab configuration in shared resource pages, entity detail tabs, empty states, dialog headers, breadcrumbs, and page titles with the new shared descriptor contract.
- [ ] Normalize translation lookup so the UI prefers entity-configured keys and uses fallback labels only as a final safety net.
- [ ] Remove remaining special-case wording where `field definitions`, `fixed values`, or `option values` still leak into generic components that should say `attributes`, `constants`, or `values` for standard presets.
- [ ] Update route/path builders and tests only where the display contract changes; avoid renaming stable URL segments unless there is a strong product reason.
- [ ] Review performance of tab construction:
  derive the rendered tab model once per entity-type query result and avoid repeated per-render recomputation.
- [ ] Extend the entity-type create/edit flow in `EntitiesWorkspace` so resource-surface titles and related settings are manageable through entity settings, not only via hardcoded preset data.
- [ ] Reuse the existing form/dialog infrastructure:
  existing entity-type dialog tabs, validators, template-mui dialog components, and current JSON-backed form state should be extended instead of adding a separate editor screen or a duplicate component family.

### Example Safe Rendering Pattern

```ts
const resourceTabs = useMemo(() => {
  return entityType.resourceSurfaces
    ?.filter((surface) => isEnabledComponentConfig(entityType.components[surface.capability]))
    .map((surface) => ({
      value: surface.key,
      label: t(surface.titleKey, surface.fallbackTitle)
    })) ?? []
}, [entityType, t])
```

## Phase 3: Update Built-In Templates and Add the Empty Template

- [ ] Update `basic` and `basic-demo` so their seeded standard entity types expose the corrected resource-tab titles through the new entity configuration contract.
- [ ] Create a new built-in metahub template `empty` with localized name `Empty` / `Пустой`.
- [ ] Ensure the `empty` template seeds only the minimum metahub shell required for a valid authoring workspace:
  branch/layout/settings as needed by the runtime contract, but no built-in metadata entity instances and no standard entity type presets referenced by the template.
- [ ] Verify metahub creation from `empty` lands the user in a working workspace where they can open `Entities` and create custom entity types or preset-based entity types manually.
- [ ] Decide whether `basic` remains the default template; do not silently switch the default unless product intent is explicit.
- [ ] Update metahub create-dialog behavior so `createOptions` reflect the selected template correctly:
  `empty` should show no optional preset toggles, and the create flow should not imply that standard metadata types will be auto-created.
- [ ] Update frontend mocks and create-dialog tests for template selection and create-options handling so the new `empty` template is covered explicitly.

### Template Safety Rules

- Keep seed data minimal and deterministic.
- Avoid hidden side effects during template application.
- Do not introduce template-specific code branches in runtime pages when the same result can be produced by seed data alone.

### Example Minimal Empty Template Direction

```ts
export const emptyTemplate: MetahubTemplateManifest = {
  $schema: 'metahub-template/v1',
  codename: 'empty',
  version: '0.1.0',
  minStructureVersion: '0.1.0',
  name: vlc('Empty', 'Пустой'),
  description: vlc(
    'Blank metahub template without predefined metadata entity types',
    'Пустой шаблон метахаба без заранее созданных типов метаданных'
  ),
  presets: [],
  seed: {
    layouts: [...minimalShellLayouts],
    layoutZoneWidgets: {...minimalShellWidgets},
    settings: [...defaultShellSettings]
  }
}
```

## Dependencies and Coordination Notes

- Frontend resource rendering depends on the shared resource-surface contract being designed first.
- Template refresh should land before generator and docs rewrites so tests and screenshots are based on the final seeded shape.
- Fixture regeneration depends on the updated Playwright flows.
- Documentation should be rewritten only after the product labels and flows are stable enough to avoid immediate screenshot churn.

## Potential Challenges

- Existing dynamic visibility logic in `SharedResourcesPage.tsx` may create false confidence:
  visible tabs are dynamic, but names and semantics are not yet fully entity-defined.
- The current backend and template validators use strict UI-config object schemas, so any extension of `EntityTypeUIConfig` requires synchronized changes across shared types, Zod schemas, template validation, mocks, and tests.
- Standard kinds may still leak old semantics through i18n namespaces, tests, helper names, and docs generators even after frontend labels are corrected.
- The new `empty` template can accidentally become invalid if the platform requires hidden baseline resources for navigation, migrations, or publication flows. That minimum contract must be verified explicitly.
- Fixture generators can mask product drift if they seed data through direct API shortcuts rather than following realistic product sequences.
- Docs rewrite scope is large; without an explicit inventory, it is easy to update only platform pages and miss guides, architecture pages, and screenshot generators.

## Modern Testing Notes

- Playwright current guidance still favors isolated browser contexts per test, reusable fixtures for setup, user-facing locators, and trace-based debugging instead of relying only on screenshots for failures.
- Visual screenshot assertions should be generated and compared in a stable environment, because host rendering differences still affect baselines.
- The existing repository structure already matches that direction, so the right plan is to strengthen the current Playwright stack instead of introducing another browser framework.
