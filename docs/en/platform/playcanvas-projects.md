# PlayCanvas Projects

PlayCanvas projects are design-time records stored inside a metahub branch. They let a connected `@universo-react/playcanvas-editor-frontend` package own project metadata without putting scenes, assets, or generated artifacts into `rel_metahub_packages.config`.

## Storage Model

-   Project, scene, asset, script, binding, generated-artifact, compatibility, and publication-manifest metadata lives in branch system tables named `_mhb_playcanvas_*`.
-   Small local scene payloads and generated `.js` / `.mjs` artifacts live under the separate `playcanvas-projects/...` file namespace.
-   The package attachment config may store only display settings and `playcanvasProject.defaultProjectId`.
-   Package detach does not delete PlayCanvas projects. Deleting a project removes its PlayCanvas project files.

## Resources UI

Open **Resources → Packages**, connect **PlayCanvas Editor**, then use the PlayCanvas projects panel under the package list.

The panel supports:

-   listing projects and health counts;
-   creating a project;
-   selecting a default project for the package attachment;
-   deleting a project with confirmation.

Project creation asks only for a user-facing project name. The internal codename is generated by the backend.

## Adapter API

The backend exposes storage-model endpoints and the first manager-only Editor bridge command endpoint:

-   project CRUD under `/metahub/{metahubId}/playcanvas/projects`;
-   scene metadata under `/projects/{projectId}/scenes`;
-   asset metadata and asset-scoped file reads/writes under `/projects/{projectId}/assets`;
-   script asset, scene binding, generated artifact, publish, and export endpoints.
-   typed Editor bridge commands at `/metahub/{metahubId}/playcanvas/editor-bridge/commands`.
-   minimal Editor compatibility REST under `/metahub/{metahubId}/playcanvas/editor-compatible/projects/{projectId}` for `config`, scene list/read/save, asset summaries, scoped settings, and explicit cloud-only no-op descriptors.

All routes require metahub management permission and use no-store responses for mutable authoring state. File writes are limited to small JSON/JS/MJS payloads, strict base64 bodies, MIME allow-listing, project-local paths, and optional SHA-256 content assertions. Asset-scoped file routes validate that `{assetId}` already points at the requested file reference.

The current bridge and compatibility slices support selected project loading, scene list/read/save, minimal JSON asset metadata, scoped settings, bridge capabilities, close, dirty-state messages, and single-user ShareDB-compatible snapshot persistence for the upstream Editor UI. They are bounded Universo-hosted authoring surfaces, not PlayCanvas Cloud API parity, and they do not provide durable ShareDB operation-log history or collaborative editing.

## Snapshots and Copy

Metahub snapshot export includes the `playcanvasProjects` section when projects exist. Local project files are bundled as small base64 payloads with checksums. Import restores project metadata and local files, remaps project-local IDs and file roots, and snapshots without `playcanvasProjects` still import as earlier-format snapshots.

Metahub copy clones branch metadata and copies the `playcanvas-projects/...` file tree. Metahub delete and failed import cleanup remove PlayCanvas project file trees alongside module source files.

## Publication Runtime

Published application sync uses normalized `playcanvasRuntimeManifests`, not live authoring files. Runtime manifests participate in publication hashing and persist to the application runtime table `_app_playcanvas_manifests`.

Published applications should read immutable hashed runtime assets or stored runtime manifests. In the current bounded JSON/JS/MJS slice, local PlayCanvas files are projected into self-contained `data:` runtime URLs with separate SHA-256 hashes; runtime manifests must not expose mutable `playcanvas-projects/...` authoring paths.
