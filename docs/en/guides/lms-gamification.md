# LMS Gamification

![Object records used by LMS gamification settings and achievements](../.gitbook/assets/entities/object-records.png)

The canonical LMS fixture models gamification as configuration and runtime data, not as a hardcoded LMS module.
The same generic primitives can be reused by another metahub that needs points, badges, or leaderboard-like mechanics.

## Metadata Model

The LMS template defines these Objects:

- `GamificationSettings` stores application or workspace gamification rules.
- `PointAwardRules` stores deterministic point award policies such as completed modules, accepted assignments, and manual adjustments.
- `PointTransactions` stores operational point changes and uses workflow actions for approval and reversal.
- `BadgeDefinitions` stores badge metadata and issue criteria.
- `BadgeIssues` stores badge issue records and uses workflow actions for issue and revoke flows.
- `LeaderboardSnapshots` stores calculated leaderboard rows that reports and dashboard widgets can read.

The template also defines the `PointSourceType` enumeration for course, track, assignment, training event, certificate, and manual point sources.
Application constants such as `GamificationEnabled` and `DefaultPointAward` keep app-level defaults in Sets instead of frontend code.

## Workflow And Posting

`PointTransactions` is a transactional Object.
The `ApprovePointAdjustment` and `ReversePointAdjustment` workflow actions are stored in Object metadata and are resolved by the generic runtime workflow service.
Approved point transactions can run `PointTransactionPostingScript`, which appends movement facts to `PointsLedger`.

`BadgeIssues` follows the same metadata-driven pattern with `IssueBadge` and `RevokeBadge` actions.
Runtime permissions remain fail-closed: action buttons appear only when the resolved capability policy explicitly allows them, and backend workflow mutations remain the source of truth.

## Reports

Gamification output is exposed through saved report definitions:

- `Leaderboard` reads `LeaderboardSnapshots` and can aggregate `TotalAwardedPoints`.
- `Achievements` reads `BadgeIssues`.

Existing `detailsTable`, CSV export, and `report.aggregation` stat-card widgets render these reports.
No LMS-specific leaderboard widget is required for the current fixture.

## Fixture Contract

The committed LMS snapshot is generated by Playwright and covered by unit and E2E checks.
The contract verifies seeded settings, point rules, point transactions, badge definitions, badge issues, leaderboard rows, workflow metadata, reports, and deterministic point totals.
