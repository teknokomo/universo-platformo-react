# Макеты приложения

Макеты приложения - это рантайм-копии макетов, опубликованных из метахаба, а также макеты, созданные прямо в приложении.
Они управляются в панели приложения: Приложения -> Макеты.
Панель приложения теперь переиспользует ту же общую поверхность проектирования, что и детали макета в метахабе, поэтому зоны виджетов, действия строк и карточно-списковое поведение остаются единообразными между проектированием и настройкой на стороне приложения.

![Детали макета с виджетом](../.gitbook/assets/quiz-tutorial/layout-quiz-widget.png)

## Источники

У каждого макета есть источник:

-   Метахаб: создан при синхронизации схемы коннектора из снимка публикации.
-   Приложение: создан или скопирован внутри панели управления приложением.

Источник показывается на карточках и в списковых представлениях, чтобы администратор отличал унаследованную конфигурацию от локальной.

## Локальные изменения

Администратор может:

-   создавать общие и привязанные к сущности макеты приложения;
-   копировать макет метахаба в отдельный макет приложения;
-   делать макет основным для его области;
-   включать и выключать макеты;
-   выключать, добавлять и настраивать виджеты макета.

Удаление макета метахаба в приложении не выполняет физическое удаление. Приложение помечает его как исключённый, выключенный и не основной, чтобы следующая синхронизация не восстановила его молча.
Макеты, созданные в приложении, удаляются через стандартный путь мягкого удаления приложения.

## Синхронизация

Когда синхронизация коннектора импортирует новую публикацию метахаба:

-   чистые макеты метахаба перезаписываются новым содержимым источника;
-   локально изменённые макеты метахаба помечаются как конфликт и не перезаписываются;
-   макеты метахаба, удалённые из публикации, помечаются как удалённые из источника;
-   макеты приложения сохраняются;
-   основной макет из источника не заменяет основной макет, созданный в приложении или локально изменённый в той же области.

Модель конфликтов по умолчанию работает закрыто: она сохраняет локальную работу и показывает состояние синхронизации для последующего решения.
Диалог изменений коннектора также позволяет администратору явно выбрать перезапись, если при этой синхронизации содержимое макетов из метахаба должно заменить локальные настройки унаследованных макетов.

## Рантайм-поведение

Рантайм приложения использует только активные макеты и активные виджеты.
Неактивные макеты остаются доступны в панели управления и могут быть включены позже.

Настройки приложения выводят вкладки отдельных функций из этого активного материализованного runtime-состояния. Например, вкладка «Матрица» для трактовочной сети появляется только при наличии активного виджета `interpretationNetworkWorkspace`, а изменения сохраняются в конфигурацию этого виджета. LMS-настройки «Учебный контент» не показываются и не сохраняются для несвязанных конфигураций приложения, пока материализованное runtime-состояние не содержит соответствующую LMS-конфигурацию.

## Режимы бокового меню

Конфигурация макета включает поведение бокового меню опубликованного приложения. Администратор может включить любую комбинацию режимов:

-   Широкое меню: постоянная полная левая навигация.
-   Компактное меню: постоянная навигация только иконками с доступными подписями.
-   Выдвижное меню: drawer, который открывается кнопкой в верхней панели.

Всегда сохраняется минимум один режим. Основной режим обязан входить в список включённых режимов. По умолчанию доступны все три режима, основным остаётся широкое меню, а runtime запоминает последний выбранный пользователем режим отдельно для каждого приложения.

Цели пунктов виджета меню выбираются из runtime-разделов по человекочитаемым названиям. Во время синхронизации приложения они материализуются в UUID-backed цели разделов, поэтому переименование метахаба или изменение его codename не должно ломать меню опубликованного приложения.
