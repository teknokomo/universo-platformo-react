# Макеты, привязанные к сущностям

Макеты, привязанные к сущностям, являются overlay-макетами, а не fork-копиями.
Каждый такой макет указывает на базовый глобальный макет и хранит только отличия для одного экземпляра сущности: страницы, каталога или другого типа сущности, у которого в конструкторе включена настройка макетов.

## Модель overlay

- унаследованные виджеты остаются связаны с базовым общим макетом;
- виджеты, принадлежащие конкретной сущности, живут только внутри её scoped-макета;
- точечные переопределения хранят изменения видимости, расположения и конфигурации унаследованных виджетов;
- унаследованные виджеты при материализации в приложение сохраняют связь `source_base_widget_id`;
- сохранённые runtime-виджеты получают настоящие UUID v7.

## Создание первого scoped-макета

![Редактор макета с виджетом квиза](../.gitbook/assets/quiz-tutorial/layout-quiz-widget.png)

1. Откройте целевой metahub.
2. Перейдите в Ресурсы -> Макеты для глобального контекста или откройте экземпляр сущности, тип которой поддерживает собственные макеты.
3. Создайте макет, привязанный к сущности, и выберите базовый глобальный макет.
4. При необходимости переставьте, настройте или отключите унаследованные виджеты либо добавьте виджеты только для этой области.

## Видимость виджетов

Глобальный макет остаётся переиспользуемой базой. Scoped-макет создаётся только там, где runtime-поверхности нужна другая композиция виджетов.
Для LMS-конфигурации это означает, что графики главной страницы остаются на странице «Главная», а разделы «Каталог», «Знания», «Развитие» и «Отчёты» используют тот же shell приложения без наследования виджетов, предназначенных только для главной страницы.

Для Catalog-подобных типов сущностей scoped-макет также может владеть runtime-поведением: видимостью кнопки создания, режимом поиска и поверхностью создания/редактирования/копирования.
Для других типов сущностей применяется та же overlay-модель для композиции виджетов и поведения, которое явно открыто компонентным контрактом этого типа.

## Публикация и рантайм

![Коннекторы опубликованного приложения](../.gitbook/assets/platform/application-connectors.png)

Публикация уплощает эффективный scoped-макет в обычные runtime-строки макета и виджета.
Приложения не вычисляют overlay-логику на лету, а используют уже материализованное рантайм-состояние.
При материализации унаследованные виджеты сохраняют связь с базой, а точечные переопределения задают расположение, активность и конфигурацию в конкретной области.
Runtime приложения выбирает основной макет для текущего `scopeEntityId`; если scoped-макета нет, используется глобальный основной макет.
