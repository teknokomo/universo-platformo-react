# Обновление схем системных приложений

Фиксированные системные приложения (admin, profiles, metahubs, applications)
используют декларативные определения схем, хранящиеся в TypeScript-файлах
исходного кода. Это руководство объясняет, как безопасно изменять их схемы
базы данных и что происходит в рантайме, когда платформа обнаруживает
изменение.

## Предварительные требования

- Понимание структуры пакетов монорепозитория.
- Знакомство с манифестами `SystemAppDefinition` в backend-пакетах.
- Прочитайте [Жизненный цикл миграций системных приложений](../architecture/system-app-migration-lifecycle.md)
  для понимания архитектурного контекста этих шагов.

## Где хранятся определения

Каждое фиксированное системное приложение имеет манифест `SystemAppDefinition`
в своём backend-пакете:

| Системное приложение | Файл манифеста                                                         |
|---------------------|------------------------------------------------------------------------|
| admin               | `packages/admin-backend/base/src/platform/systemAppDefinition.ts`      |
| profiles            | `packages/profile-backend/base/src/platform/systemAppDefinition.ts`    |
| metahubs            | `packages/metahubs-backend/base/src/platform/systemAppDefinition.ts`   |
| applications        | `packages/applications-backend/base/src/platform/systemAppDefinition.ts` |

Манифест декларирует бизнес-таблицы, поля, типы данных и capabilities.
Он является единственным источником истины для формы схемы каждого
системного приложения.

## Пошагово: добавление нового поля

1. Откройте файл манифеста целевого системного приложения.
2. Найдите массив `targetBusinessTables` (или общую `const`, на которую он ссылается).
3. Добавьте новую запись поля в соответствующее определение таблицы:

```typescript
{
    codename: 'priority',
    physicalColumnName: 'priority',
    dataType: AttributeDataType.NUMBER,
    physicalDataType: 'INTEGER',
    defaultSqlExpression: '0',
    isRequired: true
}
```

4. Соберите пакет для проверки компиляции TypeScript:

```bash
pnpm --filter <package-name> build
```

5. Запустите полную сборку workspace для распространения изменений:

```bash
pnpm build
```

6. Запустите сервер. Платформа обнаружит изменение и применит его
   автоматически во время `initDatabase()`.

## Пошагово: добавление новой бизнес-таблицы

1. Добавьте новую запись в массив `targetBusinessTables` с `kind`, `codename`,
   `tableName`, `fields` и опциональным `presentation`.
2. Следуйте конвенции именования: `cat_` для каталогов, `doc_` для документов,
   `rel_` для связей, `cfg_` для настроек.
3. Если новая таблица имеет ссылки на другие таблицы в том же
   системном приложении, укажите `targetTableCodename` на поле типа `REF`.
4. Соберите и перезапустите сервер как описано выше.

## Пошагово: удаление поля или таблицы

Удаление полей или таблиц порождает деструктивные изменения. Diff-движок
сгенерирует записи `DROP_COLUMN` или `DROP_TABLE` и применит их автоматически,
так как путь обновления системных приложений передаёт `confirmedDestructive: true`.

1. Удалите поле или таблицу из `targetBusinessTables`.
2. Соберите и перезапустите. Платформа удалит соответствующую колонку или таблицу.
3. Убедитесь, что код приложения не ссылается на удалённую колонку перед деплоем.

## Что происходит в рантайме

При запуске сервера `ensureRegisteredSystemAppSchemaGenerationPlans()`
выполняется для каждого `application_like` системного приложения:

1. **Первый запуск (таблиц нет):** Полная генерация схемы через
   `SchemaGenerator.generateFullSchema()`. Baseline-миграция записывается
   в `<schema>._app_migrations`.

2. **Последующий запуск (таблицы совпадают):** Последний снапшот читается
   из `_app_migrations`. Diff вычисляется относительно текущего манифеста.
   Если изменений нет, загрузка продолжается без записи чего-либо.

3. **Схема изменилась в манифесте:** Diff-движок обнаруживает аддитивные и
   деструктивные изменения. `SchemaMigrator.applyAllChanges()` применяет
   только вычисленные DDL-изменения (не полное пересоздание). Новая запись
   миграции сохраняется в `_app_migrations` с `snapshotBefore`, `snapshotAfter`
   и списком изменений.

## Где хранятся миграции

Миграции хранятся **в базе данных**, а не в файлах. Каждая схема системного
приложения содержит таблицу `_app_migrations`:

```
admin._app_migrations
profiles._app_migrations
metahubs._app_migrations
applications._app_migrations
```

Каждая строка содержит:

| Колонка                | Содержимое                                          |
|------------------------|----------------------------------------------------|
| `name`                 | Имя миграции (напр. `baseline_admin_structure_0_1_0`) |
| `meta`                 | JSONB с `snapshotBefore`, `snapshotAfter`, `changes`, `summary` |
| `publication_snapshot` | Всегда `null` для системных приложений              |
| `_upl_created_at`      | Временная метка применения миграции                  |

## SQL-миграции vs. изменения через манифест

Системные приложения также могут иметь SQL-миграции в массиве `migrations`
(записи с `kind: 'sql'`). Они отделены от генерации схемы через манифест
и служат другой цели:

- **`pre_schema_generation`** SQL-миграции выполняются до того, как
  `SchemaGenerator` создаст таблицы. Обычно создают саму схему или
  вспомогательные функции.
- **`post_schema_generation`** SQL-миграции выполняются после создания
  таблиц. Добавляют индексы, RLS-политики, seed-данные или функции БД.
- **Изменения через манифест** (через `targetBusinessTables`) полностью
  обрабатываются diff-движком и записываются в `_app_migrations`.

## CLI-команды

CLI миграций предоставляет команды для инспекции и управления схемами
системных приложений:

```bash
# Показать план генерации схемы для всех системных приложений
pnpm migration system-app-schema-plan

# Показать план для конкретного приложения
pnpm migration system-app-schema-plan --keys=admin

# Применить планы генерации схемы к базе данных
pnpm migration system-app-schema-apply

# Выполнить bootstrap метаданных структуры (_app_objects, _app_attributes)
pnpm migration system-app-schema-bootstrap

# Запустить полную проверку здоровья платформы включая валидацию системных приложений
pnpm migration doctor
```

## Типичные ошибки

- **Забыли собрать перед перезапуском.** Сервер загружает скомпилированный JS
  из `dist/`. Если вы изменили TypeScript-исходник, но пропустили сборку,
  используется старый манифест и миграция не выполняется.
- **Ссылка на удалённую колонку в коде приложения.** Diff-движок её удалит,
  но запросы, которые всё ещё её используют, упадут в рантайме.
- **Рассинхрон `currentBusinessTables` и `targetBusinessTables`.** Держите их
  синхронизированными, если только вы не моделируете намеренную миграцию
  из одной формы в другую.
