# Учебный контент LMS

![Рабочее пространство ресурсов, используемое Учебным контентом](../.gitbook/assets/entities/resources-workspace.png)

Учебный контент превращает общий LMS-шаблон в рабочую среду, близкую к iSpring: проекты, отдельные материалы, курсы, треки, избранное, недавнее и корзина.
Архитектура остаётся metadata-driven: метахаб определяет Объекты, Наборы, Перечисления, макеты и значения по умолчанию, а основная работа с контентом выполняется в опубликованном приложении внутри каждого рабочего пространства.

## Область реализации

Текущий срез покрывает первый слой авторинга:

-   `ContentProjects` группируют учебные материалы внутри рабочего пространства.
-   `LearningResources` хранит отдельные Страницы и Ссылки, а поле `Body` редактируется как Editor.js block JSON.
-   `Courses` являются оболочками курсов с разделами и каноническими `CourseItems`.
-   `LearningTracks` являются треками на основе курсов со стадиями `TrackStages` и шагами `TrackSteps`.
-   `ContentStars`, `RecentContentViews`, `ContentAccessEntries` и `TrashEntries` моделируют разделы Recent, Starred, Shared with me и Trash.
-   `ContentProgress` хранит серверный learner progress для завершения metadata-defined страниц и ресурсов.
-   Удаление и восстановление остаются универсальным runtime-механизмом и используют optimistic `_upl_version`.

Импорт файлов, SCORM/xAPI tracking, конвертация офисных форматов, внутренние сообщения и AI-генерация пока отложены.

## Граница рабочего пространства

Проекты Learning Content не являются рабочими пространствами платформы и не являются метахабами.
Это обычные runtime-записи Объектов внутри текущего рабочего пространства приложения.
Пользователи с достаточными правами рабочего пространства могут создавать, редактировать, копировать, удалять и восстанавливать записи учебного контента в опубликованном приложении.

## Опубликованный интерфейс

LMS-шаблон использует существующие примитивы `packages/apps-template-mui`:

-   обычное боковое меню;
-   `detailsTable`;
-   `relationBuilder`;
-   `columnsContainer`;
-   управляемую видимость колонок MUI Data Grid;
-   row actions из metadata;
-   существующие CRUD-диалоги, Editor.js поля и визуальные поля источников ресурсов.

Единый раздел Учебного контента работает через универсальный datasource `records.union`, который объединяет ресурсы, курсы и учебные треки без отдельного LMS-only виджета.
Union-таблица объединяет projection columns всех настроенных targets, поэтому поле, которое есть только у курсов, треков или ресурсов, не теряется из-за того, что другой target указан первым.
Этот же слой projection превращает настроенные ссылки на проект в читаемую колонку `Project`, а сырые значения `ProjectId` остаются скрытыми от обычных таблиц и карточек.
Toolbar таблицы также использует metadata-defined target filters, поэтому пользователь может сузить тот же union-view до ресурсов, курсов или учебных треков без просмотра внутренних target codenames.
Сохранённые отчёты по Учебному контенту могут переиспользовать тот же универсальный datasource `records.union`.
Fixture включает `LearningContentSummary`, который выводит Type, Title, Status, Instructor и Project по ресурсам, курсам и трекам, а значения ссылок нормализует до читаемых подписей до выдачи отчёта или CSV export.
Тот же отчёт использует обычный Object component `Instructor` как безопасное бизнес-поле через `records.union.projectedFields`, поэтому application-level preset колонок Learning Content и records-union фильтрация/CSV export переиспользуют metadata без отдельного published-app custom-field fork.
Вкладки отчётов Course Builder и Learning Track Builder запускают сохранённые универсальные определения отчётов через `detailsTable.reportCodename`, поэтому их report widgets работают через тот же runtime reports API, что и основной раздел Reports, а не зависят от несохранённых inline-only определений.
Базовый объект `Reports` скрывает JSON-поля конфигурации, например filters, definitions и saved filters, из обычных таблиц; пользователи видят названия и категории отчётов, а не сырой report JSON.
Активные union-представления показывают действия Избранного, Доступных и обновления поля через универсальную metadata `detailsTable.rowActions`, поэтому пользователь может добавить ресурс, курс или трек в личные библиотечные представления, поделиться контентом с участником рабочего пространства или переместить контент в другой проект без открытия полной формы редактирования.
Действие Доступных остаётся server-owned: браузер отправляет только целевую строку, желаемое состояние и, если это настроено, выбранного участника рабочего пространства; backend проверяет membership и сам определяет уровень доступа по умолчанию, timestamp и настроенную связь `ContentAccessEntries` из `runtimeLibrary.shared`.
Представления Доступных используют timestamp этой же настроенной связи через универсальную projection `sharedAt`, поэтому контент, недавно добавленный в доступные текущему пользователю, показывается первым без вывода ID строк доступа или principal ID.
Recent-представления используют ту же универсальную metadata runtime library: открытие или завершение metadata-defined записи контента обновляет настроенную связь `RecentContentViews.ViewedAt`, поэтому раздел Recent наполняется реальной runtime-активностью, а не только seed-строками, и сортируется от новых просмотров к старым через универсальную projection `recentAt`.
Та же metadata `detailsTable` может показывать универсальное меню создания для union-представлений.
Основная библиотека Учебного контента использует его для Проекта, Страницы, Ссылки, Курса и Учебного трека; runtime резолвит выбранный целевой Объект и открывает существующую CRUD-форму только после загрузки его схемы.
Quiz-lite, Assignment-lite и Импорт пакета показываются в том же меню создания как отключенные metadata-defined targets с локализованными причинами отложенной реализации, поэтому авторы видят планируемый объем Learning Content, но не попадают в недоделанные workflow оценивания, импорта файлов, SCORM или xAPI.
Создание проекта открывает обычный CRUD-диалог `ContentProjects` без LMS-only runtime-кода, поэтому авторы могут создавать контейнеры проектов рабочего пространства с той же поверхности библиотеки, где создают контент.
Targets Страница и Ссылка используют типизированную metadata `createDefaults`, чтобы предварительно заполнить только безопасные редактируемые поля формы: значение перечисления `ResourceType` и draft-тип визуального `resourceSource`.
Targets Курс и Учебный трек используют тот же универсальный контракт `createDefaults` с безопасными источниками `contextPath`, поэтому настройки Learning Content на уровне приложения предварительно заполняют навигацию, завершение, формат статуса и порядок трека без передачи всего объекта настроек в форму.
Для ресурсов типа Страница metadata `resourceSource` выводит внутренний ключ источника страницы из пользовательского заголовка/названия, поэтому автор работает с обычными полями заголовка и блочного контента, а не с техническим page codename.
Runtime отбрасывает create defaults для workspace, owner, user, lifecycle, progress, `_upl_*`, скрытых, read-only, table и отсутствующих полей до передачи initial data в форму.
Course Builder и Track Builder реализуются scoped-layouts через универсальный виджет `relationBuilder`, который держит `CourseSections`, `CourseItems`, `TrackStages` и `TrackSteps` в контексте выбранной родительской записи и переиспользует существующие CRUD-диалоги, picker-ы и механизмы сортировки строк.
Поля политик курса и трека используют metadata-defined select controls на основе общих контрактов Learning Content вместо свободного текстового ввода.
Универсальный виджет `detailsTable` также умеет показывать metadata-defined предупреждения по количеству строк; outline курса использует этот контракт на пороге 100 элементов, чтобы большие курсы были заметны до публикации.

Раздел Knowledge использует те же runtime-примитивы опубликованного приложения, что и Учебный контент.
Авторы могут создавать и редактировать `KnowledgeArticles` с выбором папки, заголовком и Editor.js body через обычный CRUD-диалог, а browser flow проверяет сохранённую строку через API приложения.
Fixture contract также проверяет seeded Knowledge spaces, folders, articles и bookmarks, которые указывают на статьи, а generic runtime mutation и Learning Content trash/restore contracts дают текущую auditable lifecycle boundary для создаваемых статей.
Отдельный UI для bookmarks и permission-limited knowledge search остаются явными deferred product capabilities, а не скрытыми частично готовыми функциями.

## Создание ресурсов

`LearningResources.Body` использует общий блочный редактор Editor.js вместо ручного JSON.
`LearningResources.Source` использует универсальный form widget `resourceSource`, поэтому авторы настраивают страницы, ссылки, встраивания, видео, аудио, документы, пакеты и отложенные файловые источники через обычные контролы.
Для источников-ссылок этот же виджет показывает проверенный внешний домен до сохранения, чтобы автор мог заметить неверный или небезопасный URL без просмотра сырых данных источника.
Во время редактирования форма показывает существующий компонент `ResourcePreview`, а некорректный источник блокирует сохранение.
Backend всё равно валидирует блочный контент и источники ресурсов перед сохранением, поэтому визуальный редактор является слоем удобства, а не единственной границей безопасности.
`LearningResources.Title` также заполняет `LearningResources.Name` через универсальную form metadata до тех пор, пока автор не изменит название вручную.
Скрытый флаг ручного изменения сохраняет этот выбор, а скрытые metadata-поля не отображаются в runtime-таблицах и формах.
Опубликованные runtime-страницы строят встроенное оглавление по header-блокам Editor.js через общий контракт page blocks.
Оглавление берёт локализованные заголовки, ссылается на стабильные anchors и игнорирует невалидные не-header блоки, сохраняя backend-валидацию как границу сохранения контента.
Опубликованный dashboard теперь получает `pageBlocks` runtime-страниц напрямую из API приложения, поэтому metadata-страницы отображаются в той же MUI-поверхности, что и списки объектов.
Плеер страницы и generic learner-player widgets учитывают настройки Learning Content на уровне приложения: видимость оглавления, видимость progress header и режим завершения.
Заголовки learner player превращают кодовые имена целевых Объектов в читаемые подписи, например `Learning Resources`, вместо показа внутренних metadata names.
Текущий прогресс чтения всё ещё хранит fail-soft браузерный cache, scoped по приложению, рабочему пространству и разделу страницы, но завершение также отправляется на backend.
Backend резолвит metadata-defined контракт `learningContent.progressStore`, записывает текущего пользователя в `ContentProgress` и scoped строку по активному рабочему пространству, если runtime-приложение использует рабочие пространства.

## Настройки приложения

Глобальные значения по умолчанию для Learning Content задаются на существующей странице настроек приложения, без отдельной LMS-only панели управления.
Вкладка `Learning Content` сохраняет строгий типизированный блок настроек:

-   режим отображения библиотеки по умолчанию;
-   включённые и отложенные типы ресурсов;
-   навигацию, условие завершения и формат статуса курса по умолчанию;
-   политику порядка учебного трека по умолчанию;
-   настройки outline и progress для плеера;
-   metadata-defined progress store, который использует опубликованный page player;
-   видимость колонок по умолчанию в таблицах Learning Content.

Неподдерживаемые и неизвестные настройки удаляются перед сохранением, а дублирующиеся типы ресурсов отклоняются общим контрактом.
В runtime create defaults передаётся только curated subset этих настроек; metadata по-прежнему определяет, какие поля можно предварительно заполнять.
Видимые колонки Learning Content могут включать обычные Object components, спроецированные через `records.union.projectedFields`, например `Instructor`; такие поля остаются частью metadata метахаба и проходят фильтрацию/export через generic records-union report contracts.

Текущий scope role visibility намеренно универсальный и fail-closed.
Учебные ресурсы, курсы и треки используют `runtimeRecordAccess.mode = ownerOrShared`, поэтому read-only members видят записи, которые они создали, или записи, которыми с ними поделились через `ContentAccessEntries`.
Application role-policy settings активно поддерживают только scopes `application` и `workspace`; неподдерживаемые allow-rules для `recordOwner`, `department`, `class` и `group` понижаются перед сохранением и игнорируются runtime permission resolution.
Predicates по подразделениям, классам и группам остаются deferred до появления универсального predicate engine, который сможет одинаково применять их к таблицам, union datasources, отчётам и мутациям.

## Целостность данных

Корзина runtime использует общий lifecycle contract:

-   активные списки читают `_upl_deleted = false`;
-   корзина читает удалённые строки через `lifecycleState=deleted`;
-   восстановление требует `editContent`;
-   восстановление fail-closed при заблокированной строке, отсутствующей строке, hard-delete объекте, невалидной цели восстановления и несовпадающей `_upl_version`.
-   восстановление может дополнительно указывать валидную родительскую запись внутри рабочего пространства, если исходный проект или родительский контейнер больше нельзя использовать.
-   удалённые таблицы `records.union` показывают универсальное действие восстановления через существующий виджет details table, используя source row id и object collection id из union datasource.

Runtime-мутации используют optimistic version checks на границе API.
Адаптеры опубликованного приложения передают `_upl_version` для update, delete, copy, restore и команд сортировки строк.
Сортировка строк отправляет expected-version map для всех строк в упорядочиваемом наборе, а backend проверяет весь набор в одной транзакции до изменения порядка.
Конфликтующие по версии, дублирующиеся, отсутствующие, cross-workspace или неполные reorder-запросы отклоняются одной командой, без частичного изменения outline.

Progress Learning Content принадлежит серверу.
Браузер отправляет только intent действия, например `view`, `complete` или `recalculate`, и ссылку на целевой контент.
Он не может напрямую записать `progressPercent` или `status`: backend вычисляет эти значения на основе metadata-defined progress, completion, sequencing и aggregation rules.

Обычные runtime cells не показывают сырой JSON для структурированных значений без display label.
Такие поля должны быть скрыты, описаны явной display metadata или отрендерены специальным preview-компонентом.
