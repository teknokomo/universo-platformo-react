# Учебный контент LMS

![Рабочее пространство ресурсов, используемое Учебным контентом](../.gitbook/assets/entities/resources-workspace.png)

Учебный контент превращает общий LMS-шаблон в рабочую среду, близкую к iSpring: проекты, отдельные материалы, курсы, треки, избранное, недавнее и корзина.
Архитектура остаётся metadata-driven: метахаб определяет Объекты, Наборы, Перечисления, макеты и значения по умолчанию, а основная работа с контентом выполняется в опубликованном приложении внутри каждого рабочего пространства.

## Область реализации

Текущий срез покрывает первый слой авторинга:

- `ContentProjects` группируют учебные материалы внутри рабочего пространства.
- `LearningResources` хранит отдельные Страницы и Ссылки, а поле `Body` редактируется как Editor.js block JSON.
- `Courses` являются оболочками курсов с разделами и каноническими `CourseItems`.
- `LearningTracks` являются треками на основе курсов со стадиями `TrackStages` и шагами `TrackSteps`.
- `ContentStars`, `RecentContentViews`, `ContentAccessEntries` и `TrashEntries` моделируют разделы Recent, Starred, Shared with me и Trash.
- `ContentProgress` хранит серверный learner progress для завершения metadata-defined страниц и ресурсов.
- Удаление и восстановление остаются универсальным runtime-механизмом и используют optimistic `_upl_version`.

Импорт файлов, SCORM/xAPI tracking, конвертация офисных форматов, внутренние сообщения и AI-генерация пока отложены.

## Граница рабочего пространства

Проекты Learning Content не являются рабочими пространствами платформы и не являются метахабами.
Это обычные runtime-записи Объектов внутри текущего рабочего пространства приложения.
Пользователи с достаточными правами рабочего пространства могут создавать, редактировать, копировать, удалять и восстанавливать записи учебного контента в опубликованном приложении.

## Опубликованный интерфейс

LMS-шаблон использует существующие примитивы `packages/apps-template-mui`:

- обычное боковое меню;
- `detailsTable`;
- `relationBuilder`;
- `columnsContainer`;
- управляемую видимость колонок MUI Data Grid;
- row actions из metadata;
- существующие CRUD-диалоги, Editor.js поля и визуальные поля источников ресурсов.

Единый раздел Учебного контента работает через универсальный datasource `records.union`, который объединяет ресурсы, курсы и учебные треки без отдельного LMS-only виджета.
Course Builder и Track Builder реализуются scoped-layouts через универсальный виджет `relationBuilder`, который держит `CourseSections`, `CourseItems`, `TrackStages` и `TrackSteps` в контексте выбранной родительской записи и переиспользует существующие CRUD-диалоги, picker-ы и механизмы сортировки строк.
Поля политик курса и трека используют metadata-defined select controls на основе общих контрактов Learning Content вместо свободного текстового ввода.
Универсальный виджет `detailsTable` также умеет показывать metadata-defined предупреждения по количеству строк; outline курса использует этот контракт на пороге 100 элементов, чтобы большие курсы были заметны до публикации.

## Создание ресурсов

`LearningResources.Body` использует общий блочный редактор Editor.js вместо ручного JSON.
`LearningResources.Source` использует универсальный form widget `resourceSource`, поэтому авторы настраивают страницы, ссылки, встраивания, видео, аудио, документы, пакеты и отложенные файловые источники через обычные контролы.
Во время редактирования форма показывает существующий компонент `ResourcePreview`, а некорректный источник блокирует сохранение.
Backend всё равно валидирует блочный контент и источники ресурсов перед сохранением, поэтому визуальный редактор является слоем удобства, а не единственной границей безопасности.
`LearningResources.Title` также заполняет `LearningResources.Name` через универсальную form metadata до тех пор, пока автор не изменит название вручную.
Скрытый флаг ручного изменения сохраняет этот выбор, а скрытые metadata-поля не отображаются в runtime-таблицах и формах.
Опубликованные runtime-страницы строят встроенное оглавление по header-блокам Editor.js через общий контракт page blocks.
Оглавление берёт локализованные заголовки, ссылается на стабильные anchors и игнорирует невалидные не-header блоки, сохраняя backend-валидацию как границу сохранения контента.
Опубликованный dashboard теперь получает `pageBlocks` runtime-страниц напрямую из API приложения, поэтому metadata-страницы отображаются в той же MUI-поверхности, что и списки объектов.
Плеер страницы учитывает настройки Learning Content на уровне приложения: видимость оглавления, видимость progress header и режим завершения.
Текущий прогресс чтения всё ещё хранит fail-soft браузерный cache, scoped по приложению, рабочему пространству и разделу страницы, но завершение также отправляется на backend.
Backend резолвит metadata-defined контракт `learningContent.progressStore`, записывает текущего пользователя в `ContentProgress` и scoped строку по активному рабочему пространству, если runtime-приложение использует рабочие пространства.

## Настройки приложения

Глобальные значения по умолчанию для Learning Content задаются на существующей странице настроек приложения, без отдельной LMS-only панели управления.
Вкладка `Learning Content` сохраняет строгий типизированный блок настроек:

- режим отображения библиотеки по умолчанию;
- включённые и отложенные типы ресурсов;
- навигацию, условие завершения и формат статуса курса по умолчанию;
- политику порядка учебного трека по умолчанию;
- настройки outline и progress для плеера;
- metadata-defined progress store, который использует опубликованный page player;
- видимость колонок по умолчанию в таблицах Learning Content.

Неподдерживаемые и неизвестные настройки удаляются перед сохранением, а дублирующиеся типы ресурсов отклоняются общим контрактом.

## Целостность данных

Корзина runtime использует общий lifecycle contract:

- активные списки читают `_upl_deleted = false`;
- корзина читает удалённые строки через `lifecycleState=deleted`;
- восстановление требует `editContent`;
- восстановление fail-closed при заблокированной строке, отсутствующей строке, hard-delete объекте и несовпадающей `_upl_version`.
- удалённые таблицы `records.union` показывают универсальное действие восстановления через существующий виджет details table, используя source row id и object collection id из union datasource.
