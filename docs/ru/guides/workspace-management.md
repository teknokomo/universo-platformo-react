# Управление рабочими пространствами

Приложения с включёнными рабочими пространствами показывают рантайм-модель рабочих пространств прямо в опубликованном MUI shell приложения.

## Что может делать пользователь

![Раздел управления рабочими пространствами](../.gitbook/assets/platform/applications-list.png)

-   переключаться между доступными рабочими пространствами из навигационной панели приложения;
-   открывать полноценный раздел управления рабочими пространствами из меню приложения или переключателя рабочих пространств;
-   создавать общее рабочее пространство с серверной пагинацией и поиском;
-   приглашать активных участников приложения по email и назначать рантайм-роль рабочего пространства;
-   удалять участников из общего рабочего пространства;
-   делать рабочее пространство текущим рабочим пространством по умолчанию.
-   открывать настройки рабочего пространства для тех настроек, которые администратор приложения разрешил переопределять на уровне рабочего пространства.

## Почему это важно для LMS

LMS MVP использует рабочие пространства как границу совместной работы.
Например, одна команда преподавателей может работать в одном рабочем пространстве, а другая команда класса — в другом, и при этом рантайм-строки изолируются backend-предикатами с учётом рабочего пространства.

## Текущие роли

-   `owner`
-   `member`

Эти роли хранятся в схеме приложения и отделены от глобальных платформенных ролей.

## Настройки рабочего пространства

Настройки рабочего пространства — это явные переопределения поверх настроек приложения.
Они доступны только для тех ключей настроек, которые разрешены политикой переопределений рабочих пространств в приложении.
Если ключ заблокирован приложением, страница рабочего пространства не показывает редактор для него, а рантайм использует значение приложения.

Владельцы рабочего пространства могут менять разрешённые настройки своего рабочего пространства.
Администраторы приложения могут управлять настройками рабочего пространства через рантайм-API, даже если не состоят в этом рабочем пространстве.
Пользователи без прав администратора, которые не являются участниками рабочего пространства, по-прежнему получают отказ.

При копировании рабочего пространства копируются только активные переопределения, которые всё ещё разрешены текущей политикой приложения.
Так снятые с поддержки или заблокированные ключи настроек не переносятся в новые рабочие пространства.

## UI-компоненты

Текущая реализация использует:

-   `WorkspaceSwitcher`
-   `RuntimeWorkspacesPage`
-   общие карточные и списочные примитивы из `@universo-react/template-mui`

Эти компоненты живут в `packages/universo-react-apps-template-mui`. Страница рабочих пространств рендерится через существующий content slot dashboard details, поэтому опубликованные приложения сохраняют тот же заголовок, мобильную навигацию, боковое меню и стиль отображения данных, что и остальные runtime-разделы.
