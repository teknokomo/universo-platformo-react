# Гостевой доступ LMS

LMS MVP поддерживает доступ без регистрации в платформе.

## Как это работает

1. Оператор создаёт runtime row в `AccessLinks`.
2. Эта строка указывает на модуль или тест.
3. Гость открывает `/public/a/:applicationId/links/:slug`.
4. Публичный runtime запрашивает только отображаемое имя.
5. Backend создаёт строку студента с `is_guest = true` и guest session token.
6. После этого прогресс и ответы на тесты записываются в контексте этого guest student.

При включённых рабочих пространствах guest runtime работает только в workspace, которому принадлежит access link или уже созданная гостевая сессия.
Это не позволяет публичному LMS-трафику случайно уйти в чужое активное рабочее пространство.

## Поддерживаемые public endpoints

- `GET /public/a/:applicationId/links/:slug`
- `POST /public/a/:applicationId/guest-session`
- `GET /public/a/:applicationId/runtime`
- `POST /public/a/:applicationId/runtime/guest-submit`
- `POST /public/a/:applicationId/runtime/guest-progress`

## QR-коды

`qrCodeWidget` в общем MUI template рендерит SVG QR-код полностью на клиенте.
Используйте его, когда нужно раздать URL гостевого модуля или теста аудитории либо в печатных материалах.

## Модель хранения

Гостевые ученики хранятся в той же schema приложения, что и зарегистрированные.
Благодаря этому статистика, прогресс и ответы на тесты находятся в одном runtime-контуре без отдельной платформенной системы гостевых аккаунтов.
Токен гостевой сессии хранится в session storage и привязан к конкретным applicationId и slug, поэтому общий компьютер не сохраняет LMS-сессию дольше текущего сеанса браузера.

## Эксплуатационные замечания

- Public runtime доступен только для приложений, помеченных как public.
- Валидация access link проверяет активность и target routing до загрузки guest runtime.
- Публичная поверхность сознательно не показывает управление рабочими пространствами и элементы аутентифицированного runtime.
