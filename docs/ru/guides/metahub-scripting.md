# Скрипты метахаба

Система скриптов Metahub — это текущий v1-путь расширения для настраиваемого поведения во время выполнения.
Один и тот же контракт покрывает создание в интерфейсе metahub, упаковку в publication, sync в application и выполнение runtime.

## Что умеют скрипты

Сейчас скрипты покрывают четыре продуктовые задачи:

- добавлять серверную логику, которая реагирует на runtime-события;
- отдавать клиентский код для widget и интерактивного UI;
- разделять переиспользуемые helper-ы рабочего пространства ресурсов через модули `general/library` и импорты `@shared/<codename>`;
- прикреплять настраиваемую логику к рабочему пространству ресурсов, уровню metahub и entity-level поверхностям проектирования.

## Поддерживаемый контракт

| Ось | Текущий контракт |
| --- | --- |
| Тип источника | В интерфейсе включено только встроенное редактирование `embedded`. |
| Совместимость SDK | Поддерживается только `sdkApiVersion = 1.0.0`. |
| Импорты | `@universo/extension-sdk` разрешён всегда; потребительские скрипты также могут импортировать библиотеки рабочего пространства ресурсов через `@shared/<codename>`. |
| Роли | `module`, `lifecycle`, `widget` и `library`. |
| Области привязки | `general` в рабочем пространстве ресурсов только для `library`, плюс metahub, hub, catalog, set, enumeration и attribute для исполняемых потребителей. |
| Клиентский рантайм | Рантайм Browser Worker с fail-closed fallback, если Worker недоступен или выполнение вышло за лимит времени. |
| Серверный рантайм | Выполнение в пуле `isolated-vm` на стороне бэкенда приложений. |

## Порядок создания

![Диалог скриптов метахаба](../.gitbook/assets/quiz-tutorial/metahub-scripts.png)

1. Откройте вкладку «Скрипты» в рабочем пространстве ресурсов для общих библиотек или у метахаба / конкретной привязанной сущности для потребителей.
2. Создайте новый скрипт и выберите роль модуля до редактирования кода.
3. Оставьте тип источника как `embedded` и объявляйте только те возможности, которые реально нужны скрипту.
4. Экспортируйте класс, который наследует `ExtensionScript`.
5. Помечайте вызываемые методы через `@AtClient()`, `@AtServer()`, `@AtServerAndClient()` или `@OnEvent(...)`.
6. Сохраните черновик и исправьте любую fail-closed ошибку валидации до публикации.

## Роли и возможности

Выбор роли — не косметика. Он управляет разрешёнными возможностями, значениями по умолчанию и правилами раскрытия рантайма.

| Роль | Типичное использование | Важные примечания |
| --- | --- | --- |
| `widget` | Интерактивные виджеты, например виджет квиза. | Черновики `widget` по умолчанию получают клиентскую RPC-возможность, необходимую для рантайм-моста. |
| `module` | Общие модули бизнес-логики. | Используйте для рантайм-helper-ов, не связанных с `widget`. |
| `lifecycle` | Серверные хуки, реагирующие на события. | Обработчики жизненного цикла никогда не вызываются через публичный рантайм RPC. |
| `library` | Общие helper-ы рабочего пространства ресурсов. | Библиотеки работают только на импорт, должны оставаться чистыми и не могут объявлять декораторы или доступ к рантайм-контексту. |

## Контракт общих библиотек

- Скрипты `library` создаются только из вкладки «Скрипты» в рабочем пространстве ресурсов с областью привязки `general`.
- `general` и `library` неразделимы: рабочее пространство ресурсов отклоняет любую другую роль для `general`, а создание `library` отклоняется вне этой рабочей области.
- Библиотеки компилируются для разрешения зависимостей и валидации до сборки потребительских скриптов.
- Потребительские скрипты импортируют их через `@shared/<codename>` и сохраняют обычные правила привязки для своей области.
- Библиотеки не раскрываются как прямые рантайм-входы, RPC-цели или обработчики жизненного цикла.
- Удаление, смена кодового имени и циклы `@shared/*` завершаются fail-closed до того, как публикация сможет отгрузить сломанный граф зависимостей.

## Поток публикации и выполнения

1. Скрипты времени проектирования живут в хранилище метахаба и редактируются из вкладки «Скрипты».
2. Генерация публикации валидирует и компилирует общие библиотеки в топологическом порядке зависимостей, а затем сериализует метаданные скрипта, исходный код и скомпилированные рантайм-бандлы в снимок публикации.
3. Синхронизация приложения копирует опубликованный контракт потребительских скриптов в `_app_scripts`, а материал общих библиотек остаётся доступным только через скомпилированных потребителей.
4. Списковые рантайм-endpoint-ы возвращают только метаданные опубликованных потребительских скриптов и никогда не встраивают исполняемые бандлы.
5. Клиентский код загружается из отдельного endpoint-а клиентского бандла.
6. Серверные вызовы идут через маршрут вызова рантайм-скрипта и остаются под проверками возможностей.

## Семантика декораторов

- `@AtClient()` помечает метод, который выполняется в браузере.
- `@AtServer()` помечает метод, который выполняется на сервере.
- `@AtServerAndClient()` оставляет один метод доступным в обоих бандлах.
- `@OnEvent(...)` объявляет lifecycle-handler, который вызывается событиями, а не через public RPC.
- Вспомогательные методы без декораторов остаются private для класса.

## Контракт виджета квиза

В текущем сценарии квиза используется скрипт с ролью `widget`, который питает `quizWidget` на dashboard приложения.

```json
{
	"type": "quizWidget",
	"scriptCodename": "quiz-widget"
}
```

Стандартный runtime bridge ожидает клиентский метод `mount()` и серверный метод `submit()`.
Необязательная конфигурация widget может переопределить эти имена, но встроенный стартовый шаблон квиза использует значения по умолчанию.

## Правила fail-closed

- Неподдерживаемые версии SDK отклоняются на этапах редактирования, нормализации publication и загрузки runtime.
- Неподдерживаемые imports, `require()`, dynamic `import()` и `import.meta` отклоняются до bundling.
- Public runtime RPC может вызывать только non-lifecycle server methods у scripts, которые объявляют `rpc.client`.
- Если `_app_scripts` нельзя подготовить во время синхронизации, синхронизация приложения падает вместо тихого пропуска скриптов.
- Если браузер не может дать Worker runtime или выполнение зависает дольше допустимого бюджета, выполнение завершается по fail-closed-сценарию.

## Рекомендуемая последовательность поставки

1. Создайте script в metahub.
2. Прикрепите widget или другой consumer к layout.
3. Опубликуйте version, которая содержит обновлённый script.
4. Синхронизируйте или создайте linked application.
5. Проверьте runtime-поведение на `/a/:applicationId`.

Для конкретного примера, созданного через браузер, переходите к [туториалу по приложению-квизу](quiz-application-tutorial.md).
