# Чеклист ревью кода базы данных

Чеклист для проверки пулл-реквестов, затрагивающих бэкенд-доступ к базе данных, хранилища (persistence stores) или SQL-запросы.

---

## 1. Корректность уровня доступа

- [ ] Аутентифицированные обработчики маршрутов используют `getRequestDbExecutor(req, getDbExecutor())` (Tier 1, RLS-скоуп)
- [ ] Код администрирования/начальной загрузки использует `getPoolExecutor()` из `@universo/database` (Tier 2, без RLS)
- [ ] Код DDL/миграций использует `getKnex()` из `@universo/database` (Tier 3)
- [ ] Tier 3 доступ изолирован в явных DDL boundaries вроде package-local `src/ddl/`, а не смешан с route/store кодом
- [ ] Нет прямого импорта Knex в доменных обработчиках маршрутов или хранилищах

## 2. Безопасность идентификаторов

- [ ] Все динамические имена схем/таблиц/колонок используют `qSchema()`, `qTable()`, `qColumn()` или `qSchemaTable()` из `@universo/database`
- [ ] Нет шаблонной интерполяции строк (`${...}`) для идентификаторов в SQL
- [ ] Все пользовательские значения используют параметры привязки (`$1`, `$2`, ...)

## 3. Предикат активных строк

- [ ] Запросы к доменам приложений включают `_upl_deleted = false AND _app_deleted = false`
- [ ] Запросы к доменам метахабов включают `_upl_deleted = false AND _mhb_deleted = false`
- [ ] Кросс-схемные JOIN-ы используют правильный предикат для каждого алиаса (никогда не смешивать `_app_deleted` с алиасами метахабов)
- [ ] Потоки restore/delete явно ограничивают состояние строки, а не обновляют только по bare `id`

## 4. Безопасность транзакций и блокировок

- [ ] DDL-операции используют рекомендательные блокировки (`withAdvisoryLock`) для конкурентности
- [ ] Длительные транзакции устанавливают `statementTimeoutMs` через `withTransaction`
- [ ] Транзакции имеют минимальный скоуп — никаких внешних HTTP-вызовов внутри блоков транзакций

## 5. Обработка ошибок

- [ ] Используется `RETURNING` для подтверждения мутации (UPDATE/DELETE)
- [ ] Zero-row результат мутации приводит к явному fail-closed пути или исключению
- [ ] Диагностические запросы выполняются только на пути ошибки (не при каждой записи)
- [ ] Сообщения об ошибках никогда не раскрывают сырой SQL или внутренние имена таблиц клиентам

## 6. Нормализация результатов

- [ ] `queryMany<T>()` возвращает типизированный `T[]` (никогда не сырой объект)
- [ ] `queryOne<T>()` возвращает `T | null` (никогда undefined)
- [ ] Zod-схемы применяются на границах системы для валидации входных данных

## 7. Здоровье пула и соединений

- [ ] Нет утечек соединений — все заимствованные соединения возвращаются через жизненный цикл executor
- [ ] `isReleased()` проверяется перед повторным использованием executor-ов в долгоживущих контекстах
- [ ] `statement_timeout` (30с по умолчанию) устанавливается через хук `afterCreate`

## 8. Соответствие линтеру

- [ ] `node tools/lint-db-access.mjs` проходит с 0 нарушениями
- [ ] Новые исключённые пути добавлены в `EXCLUDED_PATTERNS` только с обоснованием в комментарии
- [ ] Нет вызовов `KnexClient.getInstance()` или `getKnex()` в доменном коде

## 9. Покрытие тестами

- [ ] SQL-first хранилища имеют прямые юнит-тесты (не только моки на уровне маршрутов)
- [ ] Критичные service-level контракты delete/restore/sync имеют прямые тесты, если route tests мокают сервис
- [ ] Хелперы безопасности идентификаторов протестированы с пейлоадами инъекций
- [ ] Хелперы нормализации результатов протестированы на гарантии типов (массив, null, никогда undefined)
