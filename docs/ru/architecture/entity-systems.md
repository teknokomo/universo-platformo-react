# Архитектура системы сущностей

Теперь метахабы используют единый entity-first конструктор и для платформенных пресетов, и для пользовательских типов метаданных. Хабы, Каталоги, Наборы и Перечисления больше не являются отдельными захардкоженными модулями продукта. Это встроенные пресеты типов сущностей, которые поставляются шаблонами метахабов и материализуются в том же реестре типов сущностей, что и пользовательские типы.

![Рабочее пространство типов сущностей](../.gitbook/assets/entities/entities-workspace.png)

## Основные слои

### Типы сущностей

Тип сущности определяет:

- свой `kindKey`
- локализованные метаданные представления и кода
- манифест компонентов, который включает схему, записи, фиксированные значения, значения вариантов, скрипты, макеты, рантайм-поведение и другие возможности
- настройки интерфейса: иконку, размещение в боковом меню, вкладки проектирования и метаданные ресурсных поверхностей

Платформенные пресеты и пользовательские типы проходят через одно и то же хранилище и один и тот же контур валидации.

### Экземпляры сущностей

Каждый проектный объект внутри метахаба является экземпляром сущности, привязанным к одному типу сущности. Создание, чтение, изменение, порядок, копирование, удаление, публикация и синхронизация рантайма работают через общие контракты сущностей, а реестр поведения подключает специализированные сценарии только там, где они действительно нужны для платформенных пресетов.

### Ресурсные поверхности

Рабочее пространство Ресурсов больше не хардкодит названия вкладок метаданных. Типы сущностей описывают ресурсные поверхности через `ui.resourceSurfaces`, где стабильный ключ и сегмент маршрута связываются с одной совместимой возможностью:

- `dataSchema` для атрибутов
- `fixedValues` для констант
- `optionValues` для значений

Страница общих Ресурсов отображает только те возможности, которые реально где-то включены в метахабе, но название, стабильный ключ и сегмент маршрута теперь берутся из контракта типа сущности, а не из строковой карты на уровне страницы.

![Рабочее пространство ресурсов](../.gitbook/assets/entities/resources-workspace.png)

### Шаблоны и пресеты

Встроенные шаблоны определяют, какие необязательные пресеты будут созданы при создании метахаба:

- `basic`: минимальное рабочее пространство, готовое к проектированию
- `basic-demo`: демонстрационный стартовый шаблон с seed-контентом
- `empty`: без необязательных пресетов; пользователь начинает с рабочего пространства Сущностей и создаёт типы вручную

Каждый пресет может создавать:

- одно определение типа сущности
- необязательные экземпляры по умолчанию
- общие значения метаданных по умолчанию
- макеты и виджеты

## Рантайм и поведенческий слой

Реестр поведения по-прежнему важен, но это уже слой специализации, а не модель владения. Рантайм- и проектные маршруты сначала разрешают тип сущности, а затем делегируют специализированным обработчикам только те виды объектов, которым действительно нужно поведение конкретного пресета. Пользовательские типы остаются внутри общего CRUD сущностей и конвейера публикации.

## Форма маршрутов

Ключевые проектные маршруты:

- `/metahub/{metahubId}/entity-types`
- `/metahub/{metahubId}/entity-type/{entityTypeId}`
- `/metahub/{metahubId}/entities`
- `/metahub/{metahubId}/entity/{entityId}`
- `/metahub/{metahubId}/shared-containers`
- `/metahub/{metahubId}/shared-entity-overrides`

Рабочее пространство Ресурсов использует общие контейнеры для макетов, атрибутов, констант, значений и общих скриптов. Страницы экземпляров сущностей используют ту же модель возможностей, но работают уже с одним конкретным объектом.

## Правила безопасности

- Ключи типов сущностей и коды остаются уникальными в пределах одной схемы метахаба.
- Ресурсные поверхности должны иметь уникальные ключи и уникальные совместимые возможности.
- Ресурсная поверхность отклоняется, если соответствующий компонент выключен.
- Потоки удаления завершаются закрыто, если ещё существуют зависимые экземпляры.
- Общее проектирование хранит разреженные строки переопределений вместо разрушительного дублирования общих метаданных.
