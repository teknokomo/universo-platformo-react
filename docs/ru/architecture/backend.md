# Архитектура бэкенда

Бэкенд строится вокруг `@universo/core-backend`, который поднимает Express
приложение, инициализирует общий рантайм Knex, выполняет платформенные миграции,
монтирует сервисные маршруты и раздаёт фронтенд-бандл.

## Паттерн рантайма

- Express обеспечивает HTTP-маршрутизацию и композицию middleware.
- Функциональные пакеты владеют маршрутами, сервисами и SQL-first persistence helper.
- `@universo/migrations-platform` валидирует и запускает платформенные миграции.
- `@universo/database` даёт общий singleton Knex и фабрики executor.

## Паттерн Controller–Service–Store

Доменные файлы маршрутов следуют паттерну **Controller–Service–Store**:

- **Тонкие маршруты** (`*Routes.ts`) — объявляют HTTP-метод + путь + ссылку на контроллер. Обычно 25–80 строк.
- **Контроллер** (`*Controller.ts`) — обработка запросов: парсинг параметров, вызов сервисов/стор, формирование ответа. Каждый обработчик обёрнут в `createMetahubHandler()`, который предоставляет контекст `(req, res, { metahub, executor })` с проверкой членства и маппингом ошибок.
- **Сервис** (опционально) — сложная бизнес-логика, охватывающая несколько сторов.
- **Стор** (`*Store.ts`) — SQL-запросы через `DbExecutor.query()` с параметризованными запросами (`$1`, `$2`, …).

### createMetahubHandler

Общий хелпер `createMetahubHandler(fn, options?)`:
1. Разрешает metahub из параметра маршрута `:metahubId`.
2. Проверяет членство (или суперпользователь/синтетический доступ).
3. Получает request-scoped `DbExecutor` с контекстом RLS.
4. Перехватывает ошибки: `MetahubDomainError` → структурированный JSON, Axios-like → проброс статуса, generic → 500.

### Обработка доменных ошибок

Домены бросают `MetahubDomainError(message, code, status)` для структурированных ошибок (например, `LIMIT_EXCEEDED`, `CONFLICT`). Они перехватываются `createMetahubHandler` и возвращаются как `{ error, code, …extras }`.

## Структура домена

Каждый бэкенд-домен располагается в `src/domains/<name>/` со следующей структурой:

```
routes/           — тонкие объявления маршрутов
controllers/      — обработчики запросов
services/         — бизнес-логика (опционально)
stores/           — SQL-персистенция
schemas/          — Zod-схемы валидации (опционально)
```

### Активные домены (metahubs-backend)

| Домен | Обработчики | Описание |
|-------|-------------|----------|
| metahubs | — | CRUD метахабов, участники, копирование и сводка доски |
| entities | — | Определения типов сущностей, экземпляры, метаданные, действия и события |
| shared | — | Контейнеры ресурсов, общие переопределения и общие helper-ы |
| branches | — | Жизненный цикл веток и активация |
| layouts | — | Общие макеты, виджеты и сохранение поведения отображения |
| publications | — | Публикации, синхронизация, версии, экспорт и импорт |
| scripts | — | Проектные скрипты, общие библиотеки и проверка bundle |
| settings | — | Настройки метахаба |
| templates | — | Реестр встроенных шаблонов и пресетов |
| ddl | — | Пакетные границы DDL рантайма |

### Активные домены (applications-backend)

| Домен | Обработчики | Описание |
|-------|-------------|----------|
| applications | — | CRUD приложений, участники, настройки, макеты и рабочие пространства |
| connectors | — | Жизненный цикл коннекторов, связи с публикациями и синхронизация схемы |

## Текущая доменная поверхность

Активная публичная поверхность бэкенда включает auth, onboarding, profiles,
metahubs, publications, applications, admin-потоки и поддержку OpenAPI-документации.

Это модульный бизнес-бэкенд для общих сервисов платформы.
