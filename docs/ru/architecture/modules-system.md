# Система модулей

Система модулей разделена на слои проектирования, публикации и рантайм-выполнения.
Каждый слой нормализует один и тот же общий контракт манифеста, чтобы роль, тип источника и возможности оставались согласованными.

## Поток

1. В проектном слое метахаба сохраняются область привязки, роль модуля, тип источника, объявленные возможности, метаданные привязки Common/library и физический контракт хранения исходника.
2. Движок модулей валидирует чистые общие библиотеки, разрешает импорты `@shared/<codename>`, компилирует исходник в серверные и клиентские бандлы и формирует нормализованный манифест.
3. Синхронизация публикации копирует активные потребляющие модули в рантайм-таблицы приложения с уникальностью `codename` в рамках области, а логика общих библиотек остаётся доступной через скомпилированных потребителей.
4. Конечные точки списка рантайма отдают только метаданные, а клиентский бандл загружается через отдельную конечную точку с ETag.
5. Серверное выполнение использует isolated-vm с пулом isolate-ов, мониторингом здоровья и context bridge под контролем возможностей.

## Граница хранения исходников

`sourceKind` описывает семантический тип authoring/runtime и в этом срезе остаётся `embedded`.
Физическое хранение отделено:

-   `inline`: текст исходника хранится в `_mhb_modules.source_code`;
-   `file`: `_mhb_modules.source_code` равен `NULL`, а бэкенд читает исходник из `UPL_MODULE_SOURCE_ROOT`.

Сохранённые файловые пути относительны к source root ветки метахаба и всегда начинаются с `modules/`.
Они никогда не включают id метахаба, id ветки, имя схемы, id привязанной сущности или абсолютный путь хоста.
Filesystem-boundary валидирует path traversal, скрытые сегменты, allowlist расширений, размер файла, symlink containment, атомарную запись, checksums и изоляцию branch root.

Опубликованные приложения остаются bundle-only. Бэкенд приложений получает скомпилированные server/client bundles и не должен читать design-time исходники с диска.

## Границы безопасности

-   Во v1 поддерживается только embedded source kind; физическое хранение может быть inline или file-backed.
-   Embedded-модули могут импортировать `@universo-react/extension-sdk`, а потребляющие модули могут дополнительно импортировать библиотеки рабочего пространства ресурсов через `@shared/<codename>`; остальные static imports, `require()`, dynamic `import()` и `import.meta` завершают компиляцию с ошибкой.
-   Runtime package imports вроде `@universo-react/playcanvas-engine`, `@universo-react/colyseus-client` и `@universo-react/colyseus-server` должны быть подключены к метахабу/приложению и объявлены как allowed package imports для активного runtime target. Compiler не разрешает произвольные package imports.
-   Область привязки `general` зарезервирована для `library`, а новый `library` вне рабочей области Resources отклоняется до сохранения бандлов.
-   Общие библиотеки должны оставаться чистыми: decorators, доступ к runtime ctx и исполняемые рантайм-точки входа отклоняются на этапе валидации.
-   Чувствительное к зависимостям удаление общей библиотеки, смена `codename` и циклы `@shared/*` завершаются закрыто до того, как синхронизация публикации сможет произвести рантайм-состояние.
-   Возможности работают по модели deny-by-default, а недоступные API выбрасывают явные рантайм-ошибки.
-   Клиентское выполнение требует браузерный рантайм с Worker вместо fallback на main thread.
-   Браузерный worker-рантайм отключает ambient network, nested-worker и dynamic-code globals до загрузки клиентского бандла.
-   Lifecycle dispatch пропускает модули, которые не объявляют capability lifecycle.

## Связанные поверхности

-   `@universo-react/extension-sdk` задаёт базовый класс для автора и decorators.
-   `@universo-react/modules-engine` отвечает за компиляцию, разделение бандлов, пул isolate-ов и мониторинг здоровья.
-   Рантайм-маршруты приложений отдают список, клиентский бандл и конечные точки серверного вызова для опубликованных модулей.
