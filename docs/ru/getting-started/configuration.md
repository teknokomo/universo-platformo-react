# Конфигурация

## Окружение бэкенда

Создайте локальный файл `.env` в `packages/universo-core-backend/base` и укажите как минимум параметры Supabase/PostgreSQL и аутентификации, необходимые серверной части.

Обычно сюда входят Supabase URL, anonymous key, JWT secret, настройки подключения к базе, конфигурация сессий и другие секреты конкретного развёртывания.

Минимальный backend-контракт для первого запуска:

```env
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_ANON_KEY=your_public_anon_key
SERVICE_ROLE_KEY=your_server_only_service_role_key
SUPABASE_JWT_SECRET=your_supabase_jwt_secret
BOOTSTRAP_SUPERUSER_ENABLED=true
BOOTSTRAP_SUPERUSER_EMAIL=demo-admin@example.com
BOOTSTRAP_SUPERUSER_PASSWORD=ChangeMe_123456!
```

Важные замечания:

-   `SERVICE_ROLE_KEY` является строго server-only секретом. Никогда не отдавайте его в browser bundle.
-   `BOOTSTRAP_SUPERUSER_ENABLED` по умолчанию равен `true`.
-   Когда bootstrap включён, backend во время старта создаёт или подтверждает реального Supabase auth user и назначает ему эксклюзивную глобальную роль `superuser`.
-   Demo bootstrap email и пароль предназначены только для local/dev bootstrap. Перед любым реальным развёртыванием обязательно замените их.
-   Если bootstrap email уже принадлежит существующему non-superuser аккаунту, старт завершится явной ошибкой вместо молчаливого повышения привилегий.

## Окружение фронтенда

Если нужны переопределения для фронтенда, создайте локальный `.env` в `packages/universo-core-frontend/base`.

Обычно сюда входят UI host или port-настройки, например `VITE_PORT`.

## Правила конфигурации

-   Не храните реальные секреты в системе контроля версий.
-   В отслеживаемых env-файлах оставляйте только placeholder-значения.
-   После изменения общих предпосылок конфигурации повторно запускайте root build.

## Локальный профиль Supabase

Обычные `.env` файлы могут продолжать указывать на удалённый проект Supabase. Для разработки без зависимости от сети в репозитории есть явный локальный профиль Supabase в Docker.

Предварительные требования:

-   Установите Docker Desktop или Docker Engine для вашей операционной системы.
-   Запустите Docker daemon и убедитесь, что `docker ps` выполняется без ошибок.
-   Проверьте доступность Supabase CLI через workspace, например `pnpm exec supabase --version`.

Запустите локальный стек и сгенерируйте локальные env-файлы:

```bash
pnpm supabase:local:start
```

Эта команда использует закоммиченный `supabase/config.toml`, создаёт Docker network с привязкой к `127.0.0.1`, запускает Supabase через CLI и записывает:

-   `packages/universo-core-backend/base/.env.local-supabase`
-   `packages/universo-core-frontend/base/.env.local-supabase`

Сгенерированные файлы игнорируются git. В них находятся локальные ключи Supabase, поэтому их нельзя переносить в tracked-файлы.

Генерация профиля бэкенда сохраняет полный контракт окружения. Генератор берёт `packages/universo-core-backend/base/.env` как предпочтительную основу, использует `packages/universo-core-backend/base/.env.example` как запасной вариант и создаёт минимальный сгенерированный профиль только если нет ни одного из этих файлов. После этого заменяются только локальные значения подключения Supabase/PostgreSQL, `NODE_ENV`, `UNIVERSO_ENV_TARGET` и безопасные отсутствующие значения по умолчанию. Существующие настройки, например флаги стартового суперпользователя, флаги сброса базы, ограничения частоты запросов, настройки хранилища, административные флаги и другие функциональные флаги, наследуются из базового env-файла. Значения только для удалённого Supabase вроде `DATABASE_SSL_KEY_BASE64`, `SUPABASE_JWKS_URL` и `SUPABASE_JWT_ISSUER` удаляются, чтобы локальный Supabase использовал собственное прямое подключение к Postgres и локальный JWT secret.

Это значит, что обычно можно держать настройки удалённого Supabase в стандартном `.env`, а `pnpm env:local-supabase` создаст одноразовый `.env.local-supabase` профиль на его основе и заменит только локальные учётные данные Supabase и прямое подключение к Postgres.

Перед стартом платформы запускайте doctor-проверку:

```bash
pnpm doctor:local-supabase
```

Doctor проверяет Docker, Supabase CLI, состояние Auth, REST API, доступ к service-role Admin API, прямое подключение к PostgreSQL и обязательный JWT secret. Если профиль указывает на URL удалённого Supabase, проверка падает, поэтому разрушительные локальные команды не смогут случайно сбросить удалённый проект.

Запустить приложение на локальном Supabase:

```bash
pnpm start:local-supabase
```

Эта команда запускает локальный стек Supabase перед генерацией и проверкой профиля, поэтому её можно выполнять даже когда контейнеры Supabase сейчас остановлены.

Для повседневной работы, где нужны только Postgres, Auth, REST API, service-role Admin API и Studio, используйте облегчённую команду:

```bash
pnpm start:local-supabase:minimal
```

Она выполняет ту же генерацию env и те же doctor-проверки, но поднимает Supabase без realtime, storage, imgproxy, edge runtime, logflare и vector. Полный `pnpm start:local-supabase` нужен, когда вы проверяете любой из этих сервисов.

Запустить чистую локальную пересборку и сброс базы только для локального профиля:

```bash
pnpm start:allclean:local-supabase
```

Локальная allclean-команда использует тот же порядок защиты: поднять локальный Supabase, сгенерировать локальный профиль, выполнить doctor, пересобрать проект и затем стартовать с `_FORCE_DATABASE_RESET=true` против локальной базы.

Аналогичная минимальная команда для сброса:

```bash
pnpm start:allclean:local-supabase:minimal
```

Веб-консоль локального Supabase называется Supabase Studio. По умолчанию CLI открывает её на `http://127.0.0.1:54323`. По этой ссылке можно посмотреть локальные таблицы базы данных, пользователей Auth, SQL-редактор, инструменты хранилища, если хранилище запущено, настройки API и другие инструменты администрирования локального Supabase. Само приложение использует локальный API URL `http://127.0.0.1:54321`; Studio -- это браузерная панель администрирования.

Остановить локальный Supabase без удаления данных:

```bash
pnpm supabase:local:stop
```

Удалить локальные контейнеры и тома Docker Supabase:

```bash
pnpm supabase:local:nuke
```

Браузерный E2E может использовать отдельный локальный экземпляр Supabase, чтобы тестовые сбросы не затрагивали базу ручной локальной разработки:

```bash
pnpm supabase:e2e:start:minimal
pnpm run build:e2e:local-supabase
pnpm run test:e2e:smoke:local-supabase
```

Выделенный локальный E2E-профиль использует API `http://127.0.0.1:55321`, порт базы данных `55322` и Studio `http://127.0.0.1:55323`. Используйте `pnpm supabase:e2e:start` или E2E-команды `*:local-supabase:full` только для тестов, которым нужны Storage, Realtime, Edge Functions или сервисы логирования.

Выбор источника Supabase для E2E управляется `E2E_SUPABASE_PROVIDER` и `E2E_SUPABASE_ISOLATION`. Рекомендуемое значение по умолчанию: удалённый выделенный Supabase. Повторное использование основной `.env` или локального профиля Supabase для разработки требует `E2E_ALLOW_MAIN_SUPABASE=true` и `E2E_FULL_RESET_MODE=off`.

Чтобы вернуться к удалённому проекту, используйте обычные `.env` файлы и команды `pnpm start` или `pnpm start:allclean`. Локальные команды всегда передают явные `UNIVERSO_ENV_FILE` и `UNIVERSO_FRONTEND_ENV_FILE`, а не меняют базовую конфигурацию удалённого Supabase.

## Режим окружения (NODE_ENV)

Переменная `NODE_ENV` управляет режимом окружения выполнения и включает/отключает определённые функции в зависимости от контекста развёртывания.

### Значения

| Значение | Описание |
| --- | --- |
| `development` | Режим локальной разработки. Включает опасные операции, подробные сообщения об ошибках и автогенерацию секретов для тестирования. |
| `production` | Продакшн-режим. Отключает опасные операции, минимизирует раскрытие ошибок, обеспечивает безопасность по умолчанию. |

### Рекомендуемая настройка

Для локальной разработки добавьте в `.env`:

```env
NODE_ENV=development
```

### Возможности режима разработки

Когда `NODE_ENV=development`:

-   **Сброс базы данных**: `FULL_DATABASE_RESET` и команда `start:allclean` включены
-   **Секрет сессии**: Автогенерируется, если `SESSION_SECRET` не установлен (эфемерный, только для тестирования)
-   **Детали ошибок**: Полные стеки в ответах об ошибках
-   **RLS Debug**: Опциональное подробное логирование для Row Level Security

### Требования продакшн-режима

Когда `NODE_ENV=production` (или не установлен):

-   **Сброс базы данных**: Заблокирован с явным сообщением об ошибке
-   **Секрет сессии**: `SESSION_SECRET` **обязателен** — запуск завершается ошибкой без него
-   **Защищённые cookies**: Автоматически включаются для сессионных cookies
-   **Сообщения об ошибках**: Минимальное раскрытие, без стеков в ответах

### Примечание по безопасности

Никогда не устанавливайте `NODE_ENV=development` в продакшн-окружениях. Это включит опасные операции, такие как сброс базы данных, и раскроет чувствительную информацию об ошибках.

## Опасная зона: полный сброс базы данных

Опция `FULL_DATABASE_RESET` выполняет полный сброс базы данных при запуске платформы. При включении все принадлежащие проекту схемы (`app_*`, `mhb_*`, фиксированные схемы) удаляются, все пользователи Supabase auth удаляются, и база остаётся в чистом состоянии перед выполнением миграций.

**Это действие необратимо.** Настоятельно рекомендуется создать резервную копию базы данных перед включением этой опции.

Использование:

```env
FULL_DATABASE_RESET=true
```

Меры безопасности:

-   По умолчанию `false`. Требует явного включения.
-   Отказывается работать при `NODE_ENV=production`.
-   Использует PostgreSQL advisory lock для предотвращения параллельных сбросов.
-   Валидирует каждое имя схемы перед удалением (удаляются только принадлежащие проекту схемы, `public` никогда не удаляется).
-   Требует указания `SUPABASE_URL` и `SERVICE_ROLE_KEY`.
-   Выполняет пост-проверку и завершается с ошибкой при обнаружении остаточных данных.

Типичный рабочий процесс для ручного тестирования или сброса старых данных:

1. Создайте резервную копию базы данных.
2. Установите `FULL_DATABASE_RESET=true` в `.env`.
3. Запустите платформу. Сброс выполняется перед миграциями и логикой bootstrap.
4. После завершения запуска установите `FULL_DATABASE_RESET=false`, чтобы предотвратить случайный сброс при следующем запуске.

### Альтернатива: команда start:allclean

Вместо ручной установки `FULL_DATABASE_RESET=true` можно использовать команду `start:allclean`, которая выполняет полный сброс без изменения файла `.env`:

```bash
pnpm start:allclean
```

Эта команда эквивалентна:

```bash
pnpm clean:all && pnpm install && pnpm build && cross-env _FORCE_DATABASE_RESET=true pnpm start
```

Пакет `cross-env` гарантирует корректную установку переменной окружения `_FORCE_DATABASE_RESET` на всех платформах (Windows, macOS, Linux). Эта переменная принудительно выполняет сброс базы данных независимо от значения переменной окружения `FULL_DATABASE_RESET`. Все меры безопасности (защита от production, advisory lock, валидация схем) сохраняются.

**Когда использовать `start:allclean` или `FULL_DATABASE_RESET`:**

| Сценарий                            | Рекомендуемый подход                |
| ----------------------------------- | ----------------------------------- |
| Разовый локальный сброс             | `pnpm start:allclean`               |
| Сброс в CI/CD пайплайне             | `FULL_DATABASE_RESET=true` в env    |
| Повторяющееся тестирование          | `FULL_DATABASE_RESET=true` временно |
| После случайного повреждения данных | `pnpm start:allclean`               |

**Важно**: Оба метода блокируются в production-окружениях.
